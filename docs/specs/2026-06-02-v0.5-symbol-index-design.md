## Context

v0.2 shipped the Recon stage end-to-end (`recon_artifacts`, `hunt_tasks`,
`agent_sessions`) and v0.3 shipped Hunt end-to-end (`findings`, per-task
sessions). After a successful v0.3 run, three things from
`docs/tool-contracts.md § Scope: symbols` are still missing: the tree-sitter
symbol index itself, the SQLAlchemy models over `symbols` / `call_sites` /
`entry_points`, and the agent-facing `find_definition`, `find_callers`, and
`query_entry_points` tools. v0.3 papered over this by registering only
`read_file`, `list_dir`, `grep`, `record_finding` for Hunters and explicitly
forbidding `confidence='confirmed'` in the prompt.

v0.5 is the **smallest vertical slice that proves the symbol-index runtime
works end-to-end**. It implements just enough of the index machinery to:

- parse a target repo's source files with tree-sitter,
- populate the existing `symbols` / `call_sites` / `entry_points` tables for
  the current `run_id` between Recon and Hunt,
- expose `find_definition`, `find_callers`, and `query_entry_points` as tools
  whose signatures match `docs/tool-contracts.md` exactly,
- register `find_definition` and `find_callers` for Hunters so they can
  navigate symbols (`query_entry_points` stays Trace-only, gated until v0.7),
- and exit cleanly when the index can't be built (missing grammar,
  zero-symbol parse, etc.) without bringing the run down.

The Validate, Gapfill, Dedupe, Trace, and Report stages all stay deferred.
The sandbox layer (v0.4) stays deferred. Cross-repo symbol resolution stays
deferred (v2 per `ARCHITECTURE.md § v1 scope summary`).

**Milestone numbering is provisional.** The main agent chose v0.5; the
operator is free to reorder relative to the in-flight v0.4 sandbox spec
and the planned v0.6 validate / v0.7 trace specs. Nothing in this design
hard-depends on v0.4 — symbol index can land before or after sandbox
without code conflict. If sandbox lands first, the only re-work is that
Hunt's tool list grows by `compile_and_run` at v0.4 and by
`find_definition` / `find_callers` at v0.5; the order is independent.

The goal is to surface tree-sitter grammar / parse-failure / call-resolution
problems early — while only one stage (Hunt) consumes the index, and while
no findings yet depend on reachability traces — and to extend
`flosswing scan <path>` so it walks Recon → IndexBuild → Hunt against the
existing `tests/corpus/v02_smoke/` corpus and produces an index that
contains the `greet` and `main` symbols with the `main → greet` call
edge resolved.

## Success criteria

A v0.5 implementation is done when **all** of the following are true:

1. `flosswing scan <path>` against `tests/corpus/v02_smoke/` runs
   Recon → IndexBuild → Hunt, the index-build phase completes between the
   two agent stages, and the process exits 0 on success.
2. After a successful run against `v02_smoke`, the state DB at
   `~/.flosswing/state.db` contains, in addition to everything v0.3 already
   asserts:
   - ≥1 row in `symbols` with `run_id` equal to the new run, including
     entries for `greet` (kind=`function`, file=`src/example/cli.py`,
     `line_start`/`line_end` matching the source) and `main` (same file,
     adjacent line range),
   - ≥1 row in `call_sites` with `run_id` equal to the new run, including
     the call edge for `main → greet` (the `greet(sys.argv[1])` line in
     `src/example/cli.py`), with `callee_symbol_id` resolved to the `greet`
     row,
   - 0 rows or ≥1 rows in `entry_points` — the exact count depends on the
     entry-point detection decision in § Entry-point detection below.
     `v02_smoke` is a CLI script, so if Recon's entry-point pass detects
     `cli`-kind entry points, `main` qualifies.
3. Calling `find_definition(symbol='greet')` from a registered Hunt
   session returns one `SymbolDefinition` with `file='src/example/cli.py'`,
   the correct `line_start` / `line_end`, `kind='function'`,
   `language='python'`, and `truncated=False`.
4. Calling `find_callers(symbol='greet')` from a registered Hunt session
   returns exactly one `CallSite` with `caller_symbol` resolving to the
   `main` function, `file='src/example/cli.py'`, the correct call-line
   number, a non-empty `snippet`, and `truncated=False`.
5. Refusal / error paths remain observable per the v0.3 categories. New
   index-specific failure paths:
   - If tree-sitter cannot parse a file, the failure is logged at WARN,
     the file is skipped, and IndexBuild continues. Recorded in the
     run-scratch `index_build.log`, not the state DB.
   - If IndexBuild ends with zero symbols recorded (e.g. unsupported
     language), the orchestrator finalizes the run as `errored` with a
     clear message and exits 1. Hunters do not start.
6. `ruff check .`, `mypy --strict flosswing`, and `pytest tests/unit`
   continue to pass; CI gains exactly the new top-level dependencies
   listed in § Dependencies (see open question §1 on grammar packages).
7. The existing `alembic upgrade head` round-trip in CI continues to
   pass — **schema is unchanged in v0.5**. We verified by reading
   `docs/schema.sql` that `symbols`, `call_sites`, and `entry_points`
   already exist (already created by `001_initial`); v0.5 only adds
   SQLAlchemy models over them.

## Architecture

### Module layout

```
flosswing/

  stages/

    index_build.py        (NEW; deterministic stage, no agent)

  index/                  (NEW package)
    __init__.py           (NEW)
    parser.py             (NEW; tree-sitter parser registry)
    walker.py             (NEW; repo walk + extension → language)
    extractor.py          (NEW; tree-sitter queries → rows)
    entry_points.py       (NEW; deterministic entry-point heuristics)
    build.py              (NEW; orchestrates walk → extract → write)

  tools/
    symbols.py            (NEW; find_definition, find_callers,
                                query_entry_points)

  state/
    models.py             (existing — add Symbol, CallSite, EntryPoint)

  prompts/
    system/hunt.md        (existing — document new tools)
```

### Component responsibilities

**`flosswing/index/parser.py`** Holds the tree-sitter parser registry.
One process-level dict: `dict[str, Language]` keyed by short language id
(`python`, `c`, `cpp`, `rust`, `go`, `javascript`, `typescript`, `java`).
Each value is the loaded `tree_sitter.Language` for that grammar. Parsers
are constructed lazily on first request and cached for the process
lifetime. A small façade returns `(parser, queries)` where `queries` is a
pre-compiled S-expression tree-sitter query object for that language's
definition/call extractors (see § Tree-sitter queries).

**`flosswing/index/walker.py`** Walks the repo from `ctx.repo_root`,
yielding `(path, language)` tuples. Reuses the v0.2 path-escape /
binary-detection / size-cap helpers from `tools/fs.py` rather than
re-implementing them. Honours a small built-in ignore list (`.git/`,
`node_modules/`, `target/`, `build/`, `dist/`, `__pycache__/`,
`.venv/`, `venv/`, `.tox/`, `.mypy_cache/`). Honours `.gitignore` is a
**flagged open question** (see Open Questions §5) — the v0.5 plumbing
demo can ship with built-in ignores only; gitignore support can land
later if eval shows the built-in list misses common cases.

**`flosswing/index/extractor.py`** Per-language tree-sitter queries
that, given a parsed file, yield two row generators:

```python
class SymbolRow(BaseModel):
    symbol: str                    # short name
    fully_qualified_name: str      # see § Naming below
    file: str                      # repo-relative POSIX
    line_start: int
    line_end: int
    kind: Literal["function", "method", "class", "struct",
                  "enum", "macro", "type"]
    language: str

class CallSiteRow(BaseModel):
    caller_fqn: str                # resolved later → caller_symbol_id
    callee_text: str               # raw text as written
    file: str
    line: int
    snippet: str                   # truncated; see size caps
```

The extractor only emits `SymbolRow` / `CallSiteRow` — it doesn't write
to the DB. `build.py` resolves `caller_fqn` and `callee_text` to symbol
ULIDs after the whole repo is walked, so forward references work in a
single pass.

Per-language coverage maps the contract's `kind` literal to tree-sitter
node types as follows (initial set — see Open Questions §2):

| Language     | function | method | class | struct | enum | macro | type |
|--------------|----------|--------|-------|--------|------|-------|------|
| Python       | ✓        | ✓      | ✓     |        |      |       |      |
| C            | ✓        |        |       | ✓      | ✓    | ✓     | ✓    |
| C++          | ✓        | ✓      | ✓     | ✓      | ✓    | ✓     | ✓    |
| Rust         | ✓        | ✓      |       | ✓      | ✓    | ✓     | ✓    |
| Go           | ✓        | ✓      |       | ✓      |      |       | ✓    |
| JavaScript   | ✓        | ✓      | ✓     |        |      |       |      |
| TypeScript   | ✓        | ✓      | ✓     |        | ✓    |       | ✓    |
| Java         | ✓        | ✓      | ✓     |        | ✓    |       |      |

Rows are skipped silently if a node has no name (anonymous functions),
no source range, or fails to produce a valid `line_start <= line_end`
range. Skip counts are logged to `index_build.log` for postmortem but
not surfaced to the agent.

**`flosswing/index/entry_points.py`** Deterministic post-pass over the
already-extracted symbols. Heuristics per kind:

- `cli`: a symbol named `main` (Python, Go, Rust, Java, C, C++), or a
  Python module with `if __name__ == "__main__":` at file scope plus a
  function called from inside that guard.
- `http`: decorator-based detection for common Python frameworks
  (Flask-style route decorators, FastAPI route methods, Django URL-conf
  view references), JS/TS shape matches on Express-style `app.METHOD(...)`
  calls, Go shape matches on `http.HandleFunc("path", handler)`, Java
  annotation-based detection (`@GetMapping`, `@PostMapping`, etc.). The
  v0.5 demo target (`v02_smoke`) is CLI-only so the demo path doesn't
  exercise `http` — but the heuristic is wired so non-toy targets get
  coverage.
- `exported`: a symbol whose file is reachable from a build manifest's
  declared public API (Go capitalized identifiers at package scope,
  Rust `pub fn` in `lib.rs`, Python entries in `pyproject.toml`
  `[project.scripts]` or `__all__`, package.json `bin` / `main`).
- `deserializer`: callers of known unsafe-deserialization sinks (Python's
  `yaml.load`, Python's binary-object loader from the standard library;
  JS `JSON.parse` when the argument flow isn't trivially safe — flagged
  as uncertain). This overlaps with attack-class taxonomy from v0.3;
  v0.5 only flags the *symbol* as a deserializer, not the call site.
- `ipc`: not detected in v0.5; bucket reserved for later.

How this feeds the `entry_points` table is the subject of Open
Question §3. Two viable designs are sketched there.

**`flosswing/index/build.py`** Orchestrates one full index build:

```python
async def build_index(
    *,
    run_id: str,
    recon_artifact_id: str,
    repo: Path,
    languages: set[str],            # from recon_artifacts.languages_json
    session_factory: SessionFactory,
    scratch_dir: Path,              # ~/.flosswing/runs/<run_id>/index/
) -> IndexBuildResult: ...
```

Steps, in order:

1. Open `scratch_dir / "index_build.log"` for append.
2. Walk the repo via `walker.walk(repo, languages_allowlist=languages)`.
3. For each file, parse with the matching tree-sitter parser. On parse
   error (`tree_sitter.Parser` returns a tree with `has_error=True` over
   the whole file), log + skip.
4. Run the extractor; collect `(SymbolRow, CallSiteRow)` tuples in
   memory. v1 repos are small enough that memory is not the bottleneck;
   if a target ever pushes us past a few MB of in-memory rows, the bulk
   insert switches to chunked batches — see Open Questions §4.
5. Within one `session_scope()` transaction:
   - bulk-insert all `Symbol` rows with new ULIDs; build an in-memory
     `dict[(file, fqn), symbol_id]`,
   - resolve each `CallSiteRow`'s `caller_fqn` → `caller_symbol_id` via
     that dict (rows whose caller can't be resolved are dropped with a
     log entry — caller resolution is mandatory),
   - resolve `callee_text` → `callee_symbol_id` via the same dict;
     unresolved callees keep `callee_symbol_id=NULL` per the schema's
     intent (the table allows it; the index `ix_call_sites_run_id_callee`
     handles NULL via SQLite's standard ordering),
   - bulk-insert `CallSite` rows.
6. Run `entry_points.detect(symbols, recon_artifact_id, run_id)`; bulk-
   insert `EntryPoint` rows in the same transaction.
7. Return `IndexBuildResult(symbols=N, call_sites=M, entry_points=K,
   files_parsed=F, files_skipped=S, duration_ms=...)`.

No agent runs here — this is a deterministic stage. We **do not** insert
an `agent_sessions` row; that table is for agent stages only.

**`flosswing/stages/index_build.py`** Thin orchestrator-stage wrapper.
Pulls `recon_artifact_id` and `languages_json` from the just-completed
Recon row, calls `flosswing.index.build.build_index(...)`, and surfaces
an `IndexBuildResult` to the orchestrator. Distinct from
`flosswing.index.build` for the same reason `stages/recon.py` is distinct
from agent runtime: the stage module is the orchestrator's contact
point, the package does the work.

**`flosswing/tools/symbols.py`** Three tool implementations strictly per
`docs/tool-contracts.md § Scope: symbols`. The Pydantic input/output
classes are copied verbatim from the contract:

```python
class FindDefinitionInput(BaseModel):
    symbol: str
    file_hint: str | None = None
    language: str | None = None

class SymbolDefinition(BaseModel):
    symbol: str
    fully_qualified_name: str
    file: str
    line_start: int
    line_end: int
    kind: Literal["function", "method", "class",
                  "struct", "enum", "macro", "type"]
    language: str

class FindDefinitionOutput(BaseModel):
    definitions: list[SymbolDefinition]
    truncated: bool
```

Implementation:

- `find_definition(inp, ctx)`:
  ```sql
  SELECT symbol, fully_qualified_name, file, line_start, line_end,
         kind, language
    FROM symbols
   WHERE run_id = :run_id
     AND symbol = :symbol
   [AND file = :file_hint    -- if provided]
   [AND language = :language -- if provided]
   ORDER BY file, line_start
   LIMIT 50;
  ```
  Set `truncated=True` if the count hits 50; the contract doesn't define
  an upper bound, but 50 is a sane plumbing cap. If zero results AND
  the `symbols` table is empty for this `run_id`, raise the contract's
  `not_indexed` error code (this should never happen in normal
  operation per the contract — IndexBuild guarantees ≥1 symbol or fails
  the run).
- `find_callers(inp, ctx)`:
  Resolve `inp.symbol` to a `target` `SymbolDefinition` first
  (same query as `find_definition`, narrowed by `file_hint` / `language`
  if provided). If the resolution yields:
  - exactly one match → query `call_sites` joined to `symbols` (for the
    caller's FQN):
    ```sql
    SELECT s_caller.fully_qualified_name AS caller_symbol,
           cs.file, cs.line, cs.snippet
      FROM call_sites cs
      JOIN symbols s_caller ON s_caller.id = cs.caller_symbol_id
     WHERE cs.run_id = :run_id
       AND cs.callee_symbol_id = :target_id
     ORDER BY cs.file, cs.line
     LIMIT :max_results;
    ```
    Set `truncated=True` if the result count == `inp.max_results`.
    Return `FindCallersOutput(target=target, call_sites=..., truncated=...)`.
  - zero matches → raise `symbol_not_found`.
  - >1 match → raise `ambiguous_symbol` with the candidate definitions
    serialised into the error message (per the contract's wording). The
    agent can retry with `file_hint`.
- `query_entry_points(inp, ctx)`:
  ```sql
  SELECT symbol, file, line, kind, attacker_controlled_input, notes
    FROM entry_points
   WHERE run_id = :run_id
   [AND kind = :kind   -- if not 'any']
   ORDER BY kind, file, line;
  ```
  Cast `attacker_controlled_input` from `INTEGER 0|1` to `bool`. No
  errors are expected per the contract.

All three queries hit existing indices from `docs/schema.sql`:
`ix_symbols_run_id_symbol`, `ix_call_sites_run_id_callee`,
`ix_entry_points_run_id`, `ix_entry_points_kind`. No new indices needed.

### SQLAlchemy models

Three new declarative models in `state/models.py`, mapping the columns
captured from `docs/schema.sql`:

```python
class Symbol(Base):
    __tablename__ = "symbols"
    id: Mapped[str] = mapped_column(primary_key=True)
    run_id: Mapped[str] = mapped_column(ForeignKey("runs.id"))
    symbol: Mapped[str]
    fully_qualified_name: Mapped[str]
    file: Mapped[str]
    line_start: Mapped[int]
    line_end: Mapped[int]
    kind: Mapped[str]          # CHECK constraint enforced by DB
    language: Mapped[str]

class CallSite(Base):
    __tablename__ = "call_sites"
    id: Mapped[str] = mapped_column(primary_key=True)
    run_id: Mapped[str] = mapped_column(ForeignKey("runs.id"))
    caller_symbol_id: Mapped[str] = mapped_column(ForeignKey("symbols.id"))
    callee_symbol_id: Mapped[str | None] = mapped_column(
        ForeignKey("symbols.id"), nullable=True
    )
    callee_text: Mapped[str]
    file: Mapped[str]
    line: Mapped[int]
    snippet: Mapped[str]

class EntryPoint(Base):
    __tablename__ = "entry_points"
    id: Mapped[str] = mapped_column(primary_key=True)
    recon_artifact_id: Mapped[str] = mapped_column(
        ForeignKey("recon_artifacts.id")
    )
    run_id: Mapped[str] = mapped_column(ForeignKey("runs.id"))
    symbol: Mapped[str]
    file: Mapped[str]
    line: Mapped[int]
    kind: Mapped[str]
    attacker_controlled_input: Mapped[int]   # 0|1; cast in tool layer
    notes: Mapped[str]
```

The schema's CHECK constraints (`ck_symbols_kind`,
`ck_symbols_lines`, `ck_call_sites_line`, `ck_entry_points_kind`,
`ck_entry_points_attacker_input`, `ck_entry_points_line`) are
**not duplicated in Python** — they're enforced server-side by SQLite.
Application code validates inputs at the Pydantic-model boundary;
the DB is the second line of defence per the v0.3 belt-and-suspenders
pattern.

We verified `docs/schema.sql` already has the columns we need for the
v0.5 implementation. **One small flag** (Open Question §6): the schema
does not store the tree-sitter node-kind (e.g. `function_definition`
vs `function_declarator`) — only the contract's normalized `kind`
literal. v0.5 doesn't need the raw node kind, but if Validate or Trace
ever wants it for ranking, a schema addition would be required. Flagged
for the operator, not changed in this spec.

### IndexBuild placement in the pipeline

`docs/tool-contracts.md § Scope: symbols` says "The symbol index is
built once by Recon using tree-sitter and cached in SQLite." Two
readings:

A. **Inside Recon.** Recon's tool list grows by an internal
   `build_symbol_index` tool the agent calls, or Recon runs the index
   build as a deterministic post-pass in `stages/recon.py`.

B. **Between Recon and Hunt.** A separate orchestrator stage runs after
   Recon completes (using Recon's languages output) and before the first
   Hunter starts.

v0.5 implements **B**, for these reasons:

- The index build is purely deterministic. Mixing it into an agent
  stage muddies the audit boundary — the `agent_sessions` row would
  either inflate to cover non-agent work, or stay agent-only and lose
  the build's wall-clock.
- Recon's job per ARCHITECTURE.md is to identify languages, build
  commands, and trust boundaries. The index build *consumes* that
  output. Putting consumer logic inside the producer stage couples
  them unnecessarily.
- The schema places `symbols` / `call_sites` / `entry_points` under
  Recon ownership (FK to `recon_artifacts.id` on entry_points), and
  Reading B doesn't break that: the build still attributes its rows
  to the just-completed `recon_artifact_id`.

Concretely:

```python
# orchestrator.run_scan (extension)
recon_result = await stages.recon.run(...)
if recon_result.outcome != "completed" or recon_result.recon_artifact_id is None:
    finalize_errored(...); return

index_result = await stages.index_build.run(
    run_id=run_id,
    recon_artifact_id=recon_result.recon_artifact_id,
    repo=repo,
    languages=recon_result.languages,
    cfg=cfg,
    session_factory=session_factory,
)
if index_result.symbols == 0:
    finalize_errored(reason="index_build_empty"); return

hunt_result = await stages.hunt.run(...)   # v0.3 + new tools
```

The agent contract (`find_definition` / `find_callers` /
`query_entry_points` are "available to" Recon per the scope matrix) is
**not** broken by Reading B. Recon doesn't get useful answers in v0.5
because the index doesn't exist yet at Recon time, but that's also true
of the contract as written: Recon can't query its own outputs before
producing them. v0.5 leaves `find_definition` / `find_callers` out of
Recon's registered tool list for now; if a future Recon prompt wants
to query a *previous* run's index, that's a separate v2 cross-run
feature.

### Tool scoping (Hunt-side registration in v0.5)

Per `docs/tool-contracts.md` scope matrix lines 566–568:

| tool                | Recon | Hunt | Validate | Gapfill | Dedupe | Trace |
|---------------------|-------|------|----------|---------|--------|-------|
| find_definition     |   ✓   |  ✓   |    ✓     |         |        |   ✓   |
| find_callers        |       |  ✓   |    ✓     |         |        |   ✓   |
| query_entry_points  |       |      |          |         |        |   ✓   |

In v0.5 the only consumer is Hunt, so Hunt's tool registration grows
from `{read_file, list_dir, grep, record_finding}` (v0.3) to
`{read_file, list_dir, grep, record_finding, find_definition,
find_callers}`. `query_entry_points` is implemented and unit-tested but
not registered to any agent in v0.5 — it lights up when Trace lands.
This matches v0.3's pattern of implementing tools at the milestone they
land in but only registering them for stages that exist.

### Tree-sitter queries

The extractor stores S-expression queries as `.scm` files alongside
each grammar binding, one file per language. Example skeleton for
Python:

```scheme
;; flosswing/index/queries/python/definitions.scm
(function_definition
  name: (identifier) @symbol.name) @symbol.def

(class_definition
  name: (identifier) @symbol.name) @symbol.def

;; flosswing/index/queries/python/calls.scm
(call
  function: [
    (identifier) @callee.name
    (attribute attribute: (identifier) @callee.name)
  ]) @call.site
```

Each language gets a `definitions.scm` + `calls.scm` pair. These files
are checked in, versioned with the spec, and validated at module import
time (a malformed query raises immediately, not at scan time).

The naming convention for `fully_qualified_name` is per-language and
deterministic:

- Python: dotted module path + `.` + nested-scope chain
  (`pkg.mod.Class.method`). Module path derived from the repo-relative
  file path.
- Go: `package.Function` or `package.Receiver.Method`.
- Rust: `crate::module::function` or `crate::module::Struct::method`.
- Java: `package.Class.method`.
- C/C++: namespace-qualified for C++, file-qualified for C (`file.c::fn`).
- JavaScript / TypeScript: `file.js::fn` (no module-graph resolution in
  v0.5 — would require a full module resolver, which is its own
  milestone).

When the FQN can't be confidently determined, the extractor falls back
to `<file>::<short_name>` and logs the fallback. The
`fully_qualified_name` column is `NOT NULL` per the schema, so
something is always written.

### Index lifecycle and run scoping

Per `docs/schema.sql § symbols` header: "Decision: scope per-run."
v0.5 honours this strictly:

- Every row in `symbols` / `call_sites` / `entry_points` has the new
  run's `run_id`.
- We do **not** reuse a previous run's index, even if the repo hasn't
  changed.
- We do **not** clear old runs' index rows mid-scan — `DELETE FROM runs
  CASCADE` (operator-driven retention) is the only path that prunes
  them.
- The query `SELECT ... WHERE run_id = :run_id` is the single hot path
  for all three tools, and is covered by every index in the schema.

This decision is flagged for confirmation in Open Question §4
(rebuild-vs-cache trade-off).

## Data flow — one `flosswing scan <path>` run with v0.5

```
1. CLI parses args               → cli.scan(path, model?, recon_budget?, hunt_budget?)
2. config.resolve()              → auth + budgets (unchanged from v0.3)
3. orchestrator.run_scan(repo, cfg)
   a. ensure state DB exists; generate run_id; create scratch dir; insert runs row
   b. await stages.recon.run(...)                          [v0.2, unchanged]
   c. if recon.outcome != 'completed' or no recon_artifact:
        finalize errored, exit 1
   d. await stages.index_build.run(run_id, recon_artifact_id, repo,
                                   languages, cfg, session_factory)
        i.   walker.walk(repo, languages) yields (path, language) tuples
        ii.  for each file: parse with the right tree-sitter parser,
             run extractor.definitions + extractor.calls, accumulate rows
        iii. within one session_scope(): bulk-insert Symbol rows,
             resolve caller/callee FKs, bulk-insert CallSite rows
        iv.  run entry_points.detect(...); bulk-insert EntryPoint rows
        v.   return IndexBuildResult
   e. if index.symbols == 0:
        finalize errored (reason='index_build_empty'), exit 1
   f. await stages.hunt.run(run_id, repo, cfg, session_factory)
        - Hunters now have find_definition + find_callers in their tool list
        - record_finding still writes to findings
   g. finalize per v0.3 rules; print summary including index stats
4. Process exits
```

Single-writer assumption from the schema still holds — only the
orchestrator process writes. IndexBuild and Hunt run sequentially in
the same Python process. The IndexBuild transaction commits *before*
Hunt starts, so Hunt's `find_definition` / `find_callers` calls always
see a fully-populated index.

## Error and refusal handling

Inherits all v0.2 / v0.3 categories. New categories added by v0.5:

| Condition | Where caught | What's stored | Run-level effect |
|---|---|---|---|
| Tree-sitter grammar package missing for an "in-scope" language | `index.parser.get_parser()` | `index_build.log` entry | per-file skip; if the entire run has zero parsed files, fall through to `index_build_empty` below |
| Tree-sitter parse error on a specific file | `index.build.build_index` per-file loop | `index_build.log` entry | per-file skip; build continues |
| Extractor query failed on a node (rare, indicates a query bug) | `index.extractor` per-node | `index_build.log` entry + traceback (scrubbed) | per-row skip; build continues |
| IndexBuild ends with zero symbols recorded | `orchestrator.run_scan` after IndexBuild | `runs.status='errored'`, `runs.error_text='index_build_empty: no symbols extracted from N files in M languages'` | exit 1; Hunt does not start |
| Hunter calls `find_definition` on a symbol not in the index | `tools.symbols.find_definition` returns empty list with `truncated=False` | n/a | agent gets a non-error empty list, can grep instead |
| Hunter calls `find_callers` and resolution is ambiguous | `tools.symbols.find_callers` | structured `ambiguous_symbol` error returned to agent | agent retries with `file_hint` |
| Hunter calls `find_callers` and symbol is unknown | `tools.symbols.find_callers` | structured `symbol_not_found` error returned to agent | agent can grep / record at lower confidence |
| Hunter calls `find_definition` and the index doesn't exist for this run | `tools.symbols.find_definition` | structured `not_indexed` error returned to agent | should be unreachable per § IndexBuild placement; logged loudly if it ever fires |

`errors.scrub()` continues to run on every string going to
`agent_sessions.refusal_text` / `error_text` and on every line written
to stderr. The new `index_build.log` is per-run scratch (not in the
state DB) and is **also** scrubbed before being written, since it may
include extractor tracebacks that touch repo source.

## Open questions / spec inconsistencies flagged during design

1. **Language coverage in the v0.5 plumbing demo.** The spec
   recommends bundling all eight v1-list grammars
   (`tree-sitter-python`, `-c`, `-cpp`, `-rust`, `-go`,
   `-javascript`, `-typescript`, `-java`) so the milestone "feels"
   v1-complete on the index side. Alternative: ship only
   `tree-sitter-python` (since `v02_smoke` is Python-only) and add
   others as eval coverage demands. The cost difference is the build
   bundle size of the seven extra grammars (≈ a few MB each) and
   per-package operator review. **Operator decision needed.**

2. **Tree-sitter grammar version pinning strategy.** Each grammar
   package has its own release cadence, and the underlying C grammar
   inside a given package version can change node names between minor
   releases (e.g. `function_item` vs `function_definition` in Rust
   pre/post a certain release). Three options:
   - Pin to exact patch versions in `pyproject.toml` (`==X.Y.Z`),
     re-pin manually on grammar updates.
   - Pin compatible release (`~=X.Y`), accept minor-update breakage
     risk.
   - Vendor the grammar `.so` files in-tree and skip the PyPI
     packages entirely (heaviest, most stable).
   The spec recommends exact-pin (option 1) for the v0.5 demo and
   revisiting if maintenance becomes annoying. **Operator decision
   needed.**

3. **How entry-point detection feeds the `entry_points` table.** Two
   options:
   - **3a. Recon agent calls a (new) `record_entry_point` tool.** This
     keeps the agent in the loop for the (subjective) "is this an
     attacker-controlled entry point?" question. Downside: requires
     adding a tool, which is a non-trivial change to
     `docs/tool-contracts.md` (frozen contracts — change requires
     operator approval).
   - **3b. Deterministic post-pass after IndexBuild.** The
     `index/entry_points.py` heuristics in this spec. No new tool, no
     contract change. Downside: misses entry points that depend on
     non-trivial framework knowledge an LLM would catch.
   - **3c. Both.** Deterministic pass populates the baseline; Recon's
     prompt can be extended in a later milestone to add an
     `add_entry_point` tool for cases the heuristics miss.
   The spec implements 3b as the default for v0.5 (no new tool, no
   contract amendment). 3c is the medium-term target. **Operator
   decision needed.**

4. **Index lifecycle: rebuild-per-run vs. cache-keyed-by-repo-hash.**
   `docs/schema.sql` is explicit ("Decision: scope per-run"), so this
   feels closed — but parsing a large repo (tens of thousands of files)
   on every scan is wasteful when nothing changed. Options:
   - **4a. Always rebuild.** Matches schema comment. v0.5 default.
   - **4b. Cache by `(repo_path, git_commit_sha)`.** Skip the build
     if a previous run with the same SHA exists; copy rows over with
     the new `run_id`. Faster on re-scans, more code, raises the
     "old run's index was wrong, now it's also the new run's wrong"
     failure mode.
   The spec implements 4a. Flagging 4b as a possible v0.5.1 follow-up
   if scan wall-clock becomes painful.

5. **`.gitignore` support in the walker.** Built-in ignore list
   covers the common cases but misses repo-specific outputs (build
   artifacts, generated files). Three options:
   - Built-in list only (v0.5 default).
   - Honour `.gitignore` via a small library (`pathspec`); adds a dep.
   - Shell out to `git ls-files` if `.git/` exists; adds a process
     dependency but no Python dep.
   The spec recommends the first (built-in list only) for the
   plumbing demo; the third (`git ls-files`) is the likely upgrade
   path since `git` is already required to clone any target. **Open.**

6. **Schema doesn't record tree-sitter raw node kind.** The `symbols`
   table stores the contract's normalized `kind` literal
   (`function | method | class | struct | enum | macro | type`) but
   not the raw tree-sitter node type that produced it. v0.5 doesn't
   need the raw kind, but Validate / Trace ranking heuristics might.
   Flagged as a potential future schema addition; not changed in
   this spec. **No action required for v0.5; flag for v0.6+.**

7. **Parse-failure threshold for run-fatal vs. warning.** v0.5
   defaults to "build is fatal only if zero symbols emerge." A
   stricter mode ("error if > 20% of in-scope files failed to parse")
   would catch grammar-version drift before it silently degrades
   coverage. Tunable threshold flagged for the eval milestone; v0.5
   stays with the simple zero-symbols rule.
