# v0.3 — Hunt plumbing demo

**Status:** approved design, pending implementation plan
**Date:** 2026-06-02
**Replaces:** —
**Supersedes:** v0.2 (Recon plumbing demo)
**Authors:** thomas + claude

---

## Context

v0.2 shipped the Recon stage end-to-end: `flosswing scan <path>` spawns the
`claude` CLI via `claude-agent-sdk`, runs Recon against a target repo,
records one `recon_artifacts` row and ≥1 `hunt_tasks`, and audits the
session in `agent_sessions`. The state DB, the tool-registry shim, the
auth-credential machinery, refusal classification, and the error/scrub
plumbing are all in place. After a successful Recon run, several
`hunt_tasks` may sit in `status='pending'` — nothing consumes them.

v0.3 is the **smallest vertical slice that proves the Hunt stage works
end-to-end**. It implements just enough of Hunt to:

- dequeue `pending` hunt_tasks from the current run,
- spawn one Hunter agent session per task with the tools its scope permits,
- have the Hunter record ≥1 `findings` row(s) per task (or terminate cleanly
  with `outcome ∈ {completed, refused, budget_exceeded, errored}`),
- update each task's `status` and `findings_count` accordingly,
- audit every Hunter session in `agent_sessions` (one row per task),
- print a usable summary including per-task outcomes and the count of
  findings recorded, and exit with the right code.

The sandbox layer, tree-sitter symbol index, parallel Hunt concurrency,
Validate / Gapfill / Dedupe / Trace / Report stages, and per-attack-class
prompt fragments beyond a single seeded example all stay deferred. Anything
that's hard or expensive (Docker sandbox, tree-sitter grammars, asyncio
nursery with budget semaphore, refusal-rephrasing retries) is intentionally
cut.

The goal is to surface Hunt-prompt and per-task-orchestration problems
early — while the rest of the harness is still cheap to refactor — and to
extend `flosswing scan <path>` so it walks Recon → Hunt against the
existing `tests/corpus/v02_smoke/` corpus and produces a `findings` row
for the deliberate command-injection sink (the trivial CLI that
shell-passes a user string).

## Success criteria

A v0.3 implementation is done when **all** of the following are true:

1. `flosswing scan <path>` against `tests/corpus/v02_smoke/` (or any repo
   where Recon queues ≥1 hunt task) runs Recon and then Hunt, processing
   every pending task from that run, and exits 0 on success.
2. After a successful run against `v02_smoke`, the state DB at
   `~/.flosswing/state.db` contains:
   - exactly one new `runs` row with `status='completed'`,
   - exactly one `recon_artifacts` row (carried over from v0.2),
   - ≥1 `hunt_tasks` rows, **each with terminal `status`** ∈
     `{completed, refused, budget_exceeded, errored}` (no `pending` /
     `running` left over),
   - ≥1 `agent_sessions` rows with `stage='hunt'` and `task_id` set, each
     with `input_tokens > 0`, `cost_usd > 0`, terminal `outcome`,
   - ≥1 `findings` rows from at least one Hunter session (specifically,
     the `command_injection` task on the deliberate sink in
     `v02_smoke/src/example/cli.py` should produce one).
3. The CLI prints (to stdout) the run_id, the Recon summary (as in v0.2),
   a per-task line for each Hunt task (`<attack_class> <scope_hint> →
   <outcome>, N findings, T tokens`), and a roll-up footer with total
   findings, total tokens, total estimated cost.
4. Refusal and budget paths are observable per Hunt task: a refused task
   gets `hunt_tasks.status='refused'`, the corresponding `agent_sessions`
   row records `outcome='refused'` with scrubbed refusal text. A
   budget-exceeded task gets `status='budget_exceeded'` and
   `agent_sessions.outcome='budget_exceeded'`. The run as a whole still
   completes (exit 0) as long as **at least one** task completed
   successfully; if every task fails terminally, run is `errored`,
   exit 1.
5. `ruff check .`, `mypy --strict flosswing`, and `pytest tests/unit`
   continue to pass; CI gains no new external dependencies.
6. The existing `alembic upgrade head` round-trip in CI continues to
   pass — schema is unchanged in v0.3 (only one new SQLAlchemy model
   over an existing table).

## Scope

### In scope for v0.3

- New modules: `stages/hunt.py`, `prompts/system/hunt.md`,
  `prompts/attack_classes/command_injection.md`.
- New SQLAlchemy model: `Finding` (over the existing `findings` table
  defined in `001_initial.py`). Other 8 tables-without-models stay
  modelless until later milestones.
- Extension of `tools/findings.py` with `record_finding` per
  `docs/tool-contracts.md § findings (Hunt-side)`. Strict signature
  compliance.
- Tool list registered to Hunt: `read_file`, `list_dir`, `grep`,
  `record_finding`. Symbol tools (`find_callers`, `find_definition`)
  and `compile_and_run` stay deferred — see § Deferred below for
  consequences.
- Extension of `orchestrator.run_scan` to call `stages.hunt.run(...)`
  after Recon completes (only when `recon.outcome == 'completed'` and
  ≥1 task was queued).
- Sequential Hunt execution. One Hunter session at a time. No
  asyncio.gather, no Semaphore, no `--budget` concurrency cap yet.
- Per-task token budget enforced (default 200k input tokens per Hunt
  session per ARCHITECTURE.md). The existing `--token-budget` flag is
  renamed `--recon-token-budget`; new `--hunt-token-budget` flag is
  added. `--token-budget` is removed (no alias).
- Refusal **detection and surfacing per task**. No retry, no
  rephrasing pass (deferred, same as v0.2).
- Unit tests covering: `tools/findings.py` (`record_finding` happy +
  error paths), `stages/hunt.py` (dequeue / iterate / status update
  wiring with stubbed runtime), extension to `agent/runtime.py`
  outcome-classification tests if needed.
- Extension to the gated integration test in
  `tests/integration/test_recon_smoke.py` (or a new
  `test_recon_to_hunt_smoke.py`) asserting at least the
  `command_injection` task produces a finding when run against
  `v02_smoke`.

### Out of scope for v0.3 (deferred)

- `compile_and_run` and the entire sandbox layer (Docker, Firejail,
  per-language Dockerfiles). **Consequence: Hunt findings in v0.3
  cannot include `poc_result` execution results.** The
  `description_required_for_confirmed` error effectively means
  Hunters can record only `confidence ∈ {likely, speculative}` in
  v0.3. Textual `poc_code` sketches (without an executed result)
  are explicitly allowed and encouraged by the Hunt prompt — see
  § Hunt prompt.
- `find_definition`, `find_callers`, tree-sitter symbol index, the
  `symbols` / `call_sites` / `entry_points` tables.
- `query_findings`, `validate_finding`, `merge_findings`,
  `link_variant`, `record_trace`, `query_run_state`,
  `query_entry_points`.
- The Validate, Gapfill, Dedupe, Trace, Report stages.
- SQLAlchemy models for the 8 tables not touched in v0.2 or v0.3.
- Parallel Hunt execution. The `--budget N` concurrency cap from
  ARCHITECTURE.md is **not** implemented in v0.3; tasks run
  serially. When parallelism lands, it gets its own design doc.
- Per-attack-class prompt fragments beyond `command_injection.md`.
  Other 22 attack classes from the registry get a placeholder
  fallback (see § Hunt prompt below) until each is authored.
- Retry-on-refusal loop and `prompts/refusal_rephrasings/`.
- Schema-vs-models CI drift check.
- `flosswing report` and `flosswing eval` remain stubs.

### Explicit non-goals (per `ARCHITECTURE.md`)

Unchanged from v0.2: no daemon, no SaaS, no auto-disclosure, no
auto-patching, no telemetry, no cross-repo, no fuzzing framework, no
non-Anthropic providers, no write access to the target repo.

### Component responsibilities

**`stages/hunt.py`** Hunt-stage orchestration:

```python
async def run(
    *,
    run_id: str,
    repo: Path,
    cfg: Config,
    session_factory: SessionFactory,
) -> HuntStageResult: ...
```

1. SELECTs all `hunt_tasks` WHERE `run_id=? AND status='pending' ORDER
   BY priority DESC, created_at ASC`. Snapshot at start; tasks added
   mid-stage by future Gapfill don't get processed in this stage run
   (Gapfill is a separate stage anyway).
2. For each task, in order:
   a. UPDATE `hunt_tasks.status='running', started_at=now`.
   b. Load `prompts/system/hunt.md` once (per-stage, not per-task).
      Compose the per-task user prompt: `attack_class`, `scope_hint`,
      `rationale`, plus the attack-class fragment from
      `prompts/attack_classes/<attack_class>.md` if present (else a
      generic fallback line).
   c. Build tool list scoped to Hunt: `[read_file, list_dir, grep,
      record_finding]`. (Symbol tools and `compile_and_run` deferred.)
   d. INSERT `agent_sessions` row with `stage='hunt'`, `task_id=<id>`,
      `model=cfg.model`, `started_at=now`, partial fields.
   e. Await `agent.runtime.run_session(...)` with
      `token_budget=cfg.hunt_token_budget`.
   f. Classify outcome, capture tokens/cost/refusal.
   g. UPDATE `agent_sessions` row with final fields.
   h. UPDATE `hunt_tasks.status=<outcome>, finished_at=now,
      findings_count=<COUNT findings WHERE hunt_task_id=?>`.
3. Returns:
   ```python
   class HuntStageResult(BaseModel):
       tasks_processed: int
       tasks_succeeded: int     # outcome=completed
       tasks_refused: int
       tasks_budget_exceeded: int
       tasks_errored: int
       findings_total: int
   ```

**No special handling of "task completed but recorded zero findings"** —
that's a valid outcome (the Hunter looked and didn't find anything).
`status='completed'` and `findings_count=0` together mean "ran clean."

**`tools/findings.py` extension — `record_finding`**:

Signature exactly per `docs/tool-contracts.md § findings (Hunt-side)`.
Implementation responsibilities:

- Validate `attack_class` against the registry in `attack_classes.py`
  (reuse the existing validator from `add_hunt_task`); raise
  `invalid_attack_class` if not present.
- Validate that `file` resolves into `ctx.repo_root` (reuse the path
  resolver from `tools/fs.py`); raise `path_not_in_repo` otherwise.
- Validate `line_start >= 1`, `line_end >= line_start`; raise
  `line_range_invalid` otherwise.
- Enforce `description_required_for_confirmed`: if
  `confidence='confirmed'` AND (`description` is empty OR (`poc_code`
  is None AND `poc_result` is None)). **In v0.3 this means
  `confidence='confirmed'` is effectively unreachable** because there
  is no `compile_and_run` to produce `poc_result`, and there is no
  trace evidence yet. The Hunt system prompt instructs the agent to
  use `likely` / `speculative` only — see § Hunt prompt.
- Generate ULID `finding_id`.
- Write the `findings` row inside `session_scope()`. Carry through
  `hunt_task_id` from the tool-registry context (the registry knows
  which task this session is for).
- Increment `hunt_tasks.findings_count` via the same transaction.
- Return `RecordFindingOutput(finding_id=..., duplicate_of=None)`.
  Duplicate detection is a Dedupe responsibility — `duplicate_of`
  stays None in v0.3.

Truncation / size caps: `description` and `suggested_fix` have no
column-side cap in `findings` (TEXT). Application-layer cap at 64 KB
each (returned via `description_too_large` / `suggested_fix_too_large`
errors) — same shape as the existing fs.py cap.

**`state/models.py`** add `Finding`:

```python
class Finding(Base):
    __tablename__ = "findings"
    id: Mapped[str] = mapped_column(primary_key=True)
    run_id: Mapped[str] = mapped_column(ForeignKey("runs.id"))
    hunt_task_id: Mapped[str] = mapped_column(ForeignKey("hunt_tasks.id"))
    attack_class: Mapped[str]
    file: Mapped[str]
    function: Mapped[str | None]
    line_start: Mapped[int]
    line_end: Mapped[int]
    severity: Mapped[str]
    confidence: Mapped[str]
    status: Mapped[str] = mapped_column(default="pending_validation")
    title: Mapped[str]
    description: Mapped[str]
    poc_code: Mapped[str | None]
    poc_result_json: Mapped[str | None]
    suggested_fix: Mapped[str | None]
    # Dedupe/Trace columns exist in the schema but are not mapped in v0.3
    # (we don't read/write them yet). Adding them is non-breaking when
    # Dedupe/Trace land.
    created_at: Mapped[str]
```

Dedupe-managed (`primary_finding_id`, `dedupe_cluster_id`,
`dedupe_role`, `root_cause_summary`, `superseded_at`) and Trace-managed
(`reachable`) columns exist in the schema but are intentionally not
mapped — Hunt doesn't read or write them, and adding mapped fields for
columns nothing in v0.3 touches would be noise. They get mapped when
Dedupe / Trace stages land.

**`orchestrator.run_scan` extension**:

After the existing Recon block returns:

```python
if recon_result.outcome == "completed" and recon_result.hunt_tasks_queued > 0:
    hunt_result = await stages.hunt.run(
        run_id=run_id, repo=repo, cfg=cfg, session_factory=session_factory,
    )
else:
    hunt_result = HuntStageResult.skipped()
```

Run finalization rules (replacing v0.2's "Recon completed AND ≥1 task
queued → completed"):

- Recon outcome != `completed` → `runs.status='errored'`, exit 1.
- Recon `completed` but 0 tasks queued → `runs.status='errored'`,
  exit 1 (unchanged from v0.2).
- Hunt skipped (Recon errored upstream) → covered by the first rule.
- Hunt processed ≥1 tasks AND ≥1 succeeded → `runs.status='completed'`,
  exit 0.
- Hunt processed ≥1 tasks AND **zero** succeeded (all
  refused/errored/budget_exceeded) → `runs.status='errored'`, exit 1.

Per-task failures inside an otherwise-completed run are
**not** run-fatal — they're surfaced in the summary and audited but
the run still completes (exit 0). This matches how Hunt is supposed
to behave under partial failure per ARCHITECTURE.md.

**`config.py` extension**: add `hunt_token_budget: int` with the same
resolution chain as `recon_token_budget` (CLI flag overrides env
overrides default). Default: 200 000 per ARCHITECTURE.md § Agent
runtime. The existing `token_budget` field is renamed
`recon_token_budget` for clarity.

**`cli.py` extension**: rename `--token-budget` → `--recon-token-budget`
(no backward-compat alias — there is one operator, breakage cost is
known and small). Add `--hunt-token-budget INTEGER` (default 200000).

### Hunt system prompt — `prompts/system/hunt.md`

Roughly:

- States that the target repo is **untrusted input** — same
  prompt-injection caveat as Recon, but stronger because Hunt reads
  source files (not just README) and is the most exposed stage after
  Recon.
- Lists the four available tools in v0.3 (`read_file`, `list_dir`,
  `grep`, `record_finding`) and their purposes.
- Explicit instruction: `compile_and_run` and symbol-lookup tools
  **are not available in this milestone**. Do not attempt to call
  them. Do not fabricate execution results.
- Hard cap on `confidence`: use `likely` or `speculative` only. The
  prompt explains: confirmed findings require executed PoC or
  reachability trace, neither of which is available yet, so the
  contract will reject `confirmed` via
  `description_required_for_confirmed`. (Tool layer enforces too —
  belt and suspenders.)
- Encourages — but does not require — a short textual `poc_code`
  sketch demonstrating the bug, with explicit instruction NOT to
  fabricate `poc_result` execution output. The Report stage will
  surface `poc_code`-without-`poc_result` for operator review.
- Defines the success condition per task: investigate the
  `scope_hint` for the given `attack_class`. Record zero or more
  findings via `record_finding`. Then stop. Recording zero findings
  is fine if no instances are observed.
- Refusal behaviour: if the agent believes the task is unsafe (e.g.
  the scope_hint is a prompt-injection payload), it should refuse
  explicitly. We surface refusals; we don't punish them.
- Hard size cap ~4 KB so it fits cleanly in the prompt cache.

### Attack-class fragment — `prompts/attack_classes/command_injection.md`

Approx. 1–2 KB. Describes the canonical shell-passthrough sinks for
each supported language (Python's shell-invoking subprocess APIs;
Go's process-exec helpers when caller passes a shell string; the
JavaScript shell-spawning child-process helpers; etc.), the
indicators (string concatenation of untrusted input into a shell
command), and what evidence a finding should include given the v0.3
tool restrictions (file, function, line range, the offending source
snippet, brief argument-flow rationale).

This is the **only** attack-class fragment in v0.3. The other 22
classes from the registry get a generic fallback in `hunt.py`:

> No attack-class-specific guidance has been authored for
> `<attack_class>` yet. Apply general code-review principles for this
> class, lean toward `confidence='speculative'`, and stop after a
> single pass through the scope hint.

This is intentional. Authoring 23 prompt fragments at once is what
the v2/eval cycle is for; v0.3 only needs the demo path through
`command_injection` to prove the loader works.

## Data flow — one `flosswing scan <path>` run with v0.3

```
1. CLI parses args             → cli.scan(path, model?, recon_budget?, hunt_budget?)
2. config.resolve()            → auth + budgets (unchanged from v0.2)
3. orchestrator.run_scan(repo, cfg)
   a. ensure state DB exists
   b. generate run_id (ULID), create scratch dir, insert runs row
   c. await stages.recon.run(...)             [v0.2, unchanged]
   d. if recon.outcome != 'completed' OR tasks_queued == 0:
        finalize run as 'errored', exit 1, return
   e. await stages.hunt.run(run_id, repo, cfg, session_factory)
      i.   snapshot pending tasks
      ii.  for each task:
           - UPDATE hunt_tasks.status='running'
           - load hunt.md + attack_classes/<class>.md (or fallback)
           - INSERT agent_sessions row (stage='hunt', task_id=...)
           - await agent.runtime.run_session(...)
                → SDK spawns claude CLI subprocess
                → model emits tool calls
                → registry dispatches to flosswing/tools/*
                → record_finding writes to findings table
                → SDK returns usage + stop reason
           - classify outcome, capture tokens / cost / refusal
           - UPDATE agent_sessions row
           - UPDATE hunt_tasks (status, findings_count)
      iii. accumulate HuntStageResult
   f. orchestrator finalize:
      - if hunt.tasks_succeeded >= 1 → runs.status='completed', exit 0
      - else                         → runs.status='errored',   exit 1
      - UPDATE runs row, print summary
4. Process exits
```

Single-writer assumption from the schema still holds — only the
orchestrator process writes. All tool calls run in the same Python
process; per-task sessions are sequential.

## Error and refusal handling

Inherits all v0.2 categories. New categories added by Hunt:

| Condition | Where caught | What's stored | Run-level effect |
|---|---|---|---|
| Hunter refused on task X | `agent.runtime` classifies | `agent_sessions.outcome='refused'`, `refusal_text=scrub(...)`, `hunt_tasks.status='refused'` | per-task; doesn't fail the run unless *every* task fails |
| Hunter over budget on task X | orchestrator round check | `outcome='budget_exceeded'`, `hunt_tasks.status='budget_exceeded'` | per-task |
| Tool error inside Hunter session | tool wrapper | structured error returned to agent, agent keeps going | none directly; reflected in eventual outcome |
| `record_finding` called with `confidence='confirmed'` (no PoC) | `tools/findings.record_finding` | `description_required_for_confirmed` returned to agent | agent gets the error, can record at lower confidence instead |
| All tasks failed terminally | orchestrator finalize | `runs.status='errored'` | exit 1 |
| ≥1 task succeeded, others failed | orchestrator finalize | `runs.status='completed'` | exit 0; summary shows per-task outcomes |

`errors.scrub()` continues to run on every string going to
`agent_sessions.refusal_text` / `error_text` and on every line
written to stderr.

## Design decisions (resolved 2026-06-02)

1. **Budget-flag naming.** Rename `--token-budget` →
   `--recon-token-budget`; add `--hunt-token-budget`. No
   backward-compat alias for `--token-budget` — breakage cost is one
   operator. `config.py` field renames `token_budget` →
   `recon_token_budget`. Defaults: 200 000 input tokens for each.

2. **`confidence='confirmed'` reachability in v0.3.** Hunt system
   prompt actively forbids `confirmed`: tells the model to use
   `likely` or `speculative` only, with the contract-level reason
   spelled out. The tool layer continues to raise
   `description_required_for_confirmed` as a backstop. Belt and
   suspenders.

3. **Attack-class fragment loader location.** Keep the loader as a
   small private helper in `stages/hunt.py`. Lift to a shared API in
   `prompts/__init__.py` only when a second consumer (Validate /
   Gapfill) materially needs it — designing a multi-consumer API on
   one consumer's needs is the kind of premature abstraction
   `CLAUDE.md` warns against.

4. **Integration-test layout.** Extend the existing
   `tests/integration/test_recon_smoke.py` rather than fork. Single
   gated invocation; Hunt-side asserts piggyback on the Recon-side
   ones. Rename only if/when the file grows enough to justify it.

5. **Schema model coverage.** `Finding` model maps only the columns
   Hunt reads/writes in v0.3 (15 of 22). Dedupe-managed and
   Trace-managed columns stay unmapped; they get mapped when those
   stages land. Adding columns to a SQLAlchemy `Mapped` class is a
   non-breaking change.

6. **PoC code sketches without execution results.** Allowed. Hunt
   prompt encourages a short textual `poc_code` demonstrating the
   bug, with an explicit instruction NOT to fabricate `poc_result`
   output. Report stage will render `poc_code`-without-`poc_result`
   for operator review.

7. **No automatic schema-vs-models drift check yet.** Same stance as
   v0.2. The check lands with the milestone that brings all 13
   SQLAlchemy models — that's when it becomes meaningful, since a
   partial check can only catch model→schema drift, not the more
   common schema→model direction.
