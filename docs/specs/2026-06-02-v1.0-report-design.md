# v1.0 — Report stage design

**Status:** draft design, pending operator approval
**Date:** 2026-06-02
**Replaces:** —
**Supersedes:** v0.3 (Hunt plumbing demo) — the `flosswing report` CLI
stub that v0.1 introduced and v0.2/v0.3 left as `"not implemented"`
**Authors:** thomas + claude

---

## Context

v0.1 shipped the project skeleton and reserved a `flosswing report
<run_id>` CLI subcommand that prints `"not implemented"`. v0.2 wired
Recon end-to-end and populated `runs`, `recon_artifacts`, `hunt_tasks`,
and `agent_sessions`. v0.3 wired Hunt and added the first `findings`
rows. After a v0.3 scan, the operator can inspect results only by
opening `~/.flosswing/state.db` in `sqlite3` — there is no human-facing
output anywhere.

v1.0 closes that gap. It is the **smallest vertical slice that turns
SQLite state into operator-facing output**, and — because it is the
**v1-completion milestone** in the road map (Validate, Gapfill, Dedupe,
and Trace each get their own preceding plumbing milestones, in the
v0.4–v0.7 range) — also the milestone where `flosswing scan` finally
matches the ARCHITECTURE.md § Operating model promise of "runs the
pipeline, writes findings to its own state directory, generates a
report."

**The v1.0 milestone version is provisional.** This spec assumes the
v0.4–v0.7 plumbing slices for Validate, Gapfill, Dedupe, and Trace ship
in order before Report; if any of those slip past Report, the v1.0
milestone version is reassigned to whichever slice ships last and
Report ships under that version's banner. Report itself is unaffected
because the schema is unchanged and the renderer degrades gracefully on
empty Validate / Trace / Dedupe tables (see § Graceful degradation).

The goal of this milestone is to surface report-rendering and
schema-projection problems early — while the rest of the harness is
still cheap to refactor — and to give the operator a `report.md` they
can actually read after a scan. Per ARCHITECTURE.md § Stage 8, this
stage is **deterministic, no agent.** This spec contains **no agent
tools, no SDK calls, no new prompts, and no extension to
`flosswing/agent/runtime.py`**. It is a pure-Python module that reads
SQLite and renders files.

## Success criteria

A v1.0 Report implementation is done when **all** of the following are
true:

1. After any successful `flosswing scan <path>` run (one that exits 0),
   the directory `~/.flosswing/runs/<run_id>/output/` exists and
   contains, at minimum:
   - `report.md`
   - `report.json`
   - `findings/` directory (may be empty if the run produced no
     CONFIRMED findings; one subdir per CONFIRMED finding otherwise)
2. `report.md` contains the operator-facing disclaimer "**A
   FlossWing scan does not mean the target is secure.**" verbatim, in a
   prominent position (top of the document, before the findings table).
   This satisfies ARCHITECTURE.md § Threat model summary point 5.
3. `report.json` validates against the Pydantic model `ReportV1`
   defined in `flosswing/stages/report.py` (see § JSON schema below)
   and round-trips through `ReportV1.model_validate_json(...)` cleanly.
4. For every finding with `status='confirmed'` in the state DB, there
   is exactly one `findings/<finding_id>/` directory containing:
   - `finding.json` — single-finding projection of the same schema
   - `poc/` — directory; contains the PoC source files if
     `findings.poc_code` is non-null; empty (but present) otherwise
   - `trace.json` — present iff a `traces` row exists for the finding;
     omitted otherwise
   - `suggested_fix.md` — present iff `findings.suggested_fix` is
     non-null; omitted otherwise
5. `flosswing report <run_id>` re-renders an existing run's output
   directory without re-running the pipeline, exits 0 on success, and
   produces byte-identical output to the auto-render at end of scan for
   the same run_id (modulo a `rendered_at` timestamp field in
   `report.json` — see § Determinism).
6. `flosswing scan <path>` exits 0 if and only if the run completed
   successfully **and** Report rendered successfully. A render failure
   on an otherwise-successful run finalizes `runs.status='completed'`
   (the scan itself succeeded) but exits 1 with a clear "report
   rendering failed; re-run with `flosswing report <run_id>`" message.
7. `ruff check .`, `mypy --strict flosswing`, and `pytest tests/unit`
   continue to pass; CI gains no new external dependencies.
8. The existing `alembic upgrade head` round-trip in CI continues to
   pass — schema is unchanged in v1.0. Report is a pure consumer of the
   schema.

## Scope

### In scope for v1.0

- New module: `flosswing/stages/report.py`. Single file at the start;
  graduates to a `flosswing/report/` package only if it crosses ~600
  lines (see Open Questions). All schema reads, markdown rendering,
  JSON projection, and per-finding-dir writes live here in v1.0.
- New Pydantic model `ReportV1` (and its sub-models: `ReportFinding`,
  `ReportRun`, `ReportSummary`, `ReportTrace`, `ReportValidation`,
  `ReportDedupeCluster`) declared in `flosswing/stages/report.py`.
  Operator-facing — **not subject to the frozen tool-contracts.md
  guarantee**, but treated as a stable public artifact that evolves
  under a documented version field (`schema_version: "1.0"` literal in
  the JSON output).
- Extension of `flosswing/orchestrator.run_scan` to invoke
  `stages.report.render(...)` at the end of any successful scan, after
  the `runs` row has been finalized to `status='completed'`. The render
  call is the last thing the orchestrator does before printing the
  summary.
- Extension of `flosswing/cli.py`:
  - `flosswing scan` gains `--no-report` (skip end-of-scan render),
    `--format` (default `md,json`; accepts comma-separated
    `md`/`json`/`sarif`).
  - `flosswing report <run_id>` is implemented (replacing the v0.1
    stub). Accepts `--format` and a `--output-dir` override; defaults
    to `~/.flosswing/runs/<run_id>/output/`.
- New SQLAlchemy models for the tables Report reads that v0.2/v0.3 did
  not yet map: `Validation`, `DedupeCluster`, `FindingLink`, `Trace`,
  `EntryPoint`. Adds the dedupe/trace columns to the existing `Finding`
  model. All adds; no migrations.
- Unit tests covering: empty-run render (no findings), single-finding
  render, multi-finding render with severity / reachability grouping,
  CONFIRMED-vs-pending finding dir behaviour, prompt-injection-safe
  markdown rendering of `description` / `poc_code` / `suggested_fix`,
  `flosswing report <run_id>` subcommand re-render parity.
- Extension to the gated integration test asserting `report.md` and
  `report.json` exist after a v0.3 smoke run, and that
  `findings/<id>/` directories exist for any CONFIRMED finding (in
  v0.3-only mode there will be none — see § Graceful degradation; the
  test asserts a present-but-empty `findings/` dir).

### Out of scope for v1.0 (deferred)

- **SARIF output.** The `--format sarif` flag is **parsed** (so we
  don't break compatibility with the eventual v1.1 implementation) but
  v1.0 prints `"sarif: not yet implemented; tracked in v1.1"` and
  continues. The flag does not cause an error. See § SARIF stance.
- HTML output of any kind.
- Diff mode (`flosswing diff <run_a> <run_b>`) — v2 per ARCHITECTURE.md.
- `flosswing disclose` — v2 per ARCHITECTURE.md.
- Rich/colorized terminal output beyond what `click.echo` already does.
- Per-language PoC syntax highlighting in `report.md`. Markdown code
  fences with language tags are emitted (e.g. ```` ```python ````),
  but no extra rendering. Operators view the markdown wherever they
  view markdown.
- Index/landing-page generation across multiple runs. Each run renders
  in isolation. A `~/.flosswing/runs/index.md` listing all runs is a
  v1.1 nicety.
- Notification / hook system on render completion.

### Explicit non-goals (per `ARCHITECTURE.md`)

Unchanged from v0.2/v0.3: no daemon, no SaaS, no auto-disclosure, no
auto-patching, no telemetry, no upload of report artifacts anywhere, no
non-Anthropic providers, no write access to the target repo. Report
**only writes** to `~/.flosswing/runs/<run_id>/output/`.

## Architecture

### Module layout

```
flosswing/
  orchestrator.py          (existing — extend to call Report)
  cli.py                   (existing — implement `report` subcommand,
                            add --no-report and --format to `scan`)
  config.py                (existing — add report_output_dir field,
                            default ~/.flosswing/runs/<run_id>/output/)

  stages/
    recon.py               (existing)
    hunt.py                (existing)
    report.py              (NEW)

  state/
    models.py              (existing — add Validation, DedupeCluster,
                            FindingLink, Trace, EntryPoint; extend
                            Finding with dedupe/trace columns)
```

No new top-level dependencies. Stack remains exactly the list in
`CLAUDE.md § Dependency policy`. Markdown rendering is hand-rolled
string assembly — no `markdown`, `mistune`, `jinja2`, or similar lib.
JSON serialization uses Pydantic's built-in `.model_dump_json()`.

### Component responsibilities

**`stages/report.py`** — single module exposing one public entry point
and a small set of internal helpers. The public API is:

```python
def render(
    *,
    run_id: str,
    session_factory: SessionFactory,
    output_dir: Path,
    formats: frozenset[Literal["md", "json", "sarif"]],
) -> ReportResult: ...


class ReportResult(BaseModel):
    run_id: str
    output_dir: Path
    files_written: list[Path]
    findings_total: int
    confirmed_count: int
    skipped_formats: list[str]   # e.g. ["sarif"] in v1.0
    rendered_at: str             # ISO-8601 UTC
```

`render(...)` is **synchronous** — there is no IO that benefits from
asyncio here, no SDK call to await, no concurrency. Calling it from the
async orchestrator is a single `report.render(...)` line; we don't
need `asyncio.to_thread` for the scale of one run's worth of rows.

Internal helpers (all module-private):

- `_load_run_data(session, run_id) -> _RunData` — single read-only
  SQLAlchemy transaction collecting every row Report needs:
  `runs`, `recon_artifacts`, `hunt_tasks`, `findings`, `validations`,
  `traces`, `dedupe_clusters`, `agent_sessions`. Joins are done in
  Python after loading; the dataset per run is small (hundreds of
  rows at most for the corpus repos in scope).
- `_build_report_model(data) -> ReportV1` — projects the raw rows into
  the operator-facing Pydantic shape. This is where ordering decisions,
  grouping decisions, and field-by-field projections live.
- `_render_markdown(report) -> str` — pure-function string assembly.
- `_render_json(report) -> str` — `report.model_dump_json(indent=2)`.
- `_write_per_finding_dirs(report, output_dir, data) -> list[Path]` —
  iterates CONFIRMED findings, writes `finding.json`, `poc/` files,
  `trace.json` (if present), `suggested_fix.md` (if present).
- `_render_sarif_stub() -> str` — returns the literal "sarif: not yet
  implemented; tracked in v1.1" placeholder string for v1.0. Will be
  replaced by `_render_sarif(report)` in v1.1.

All helpers are unit-tested directly. The public `render()` is tested
through the integration test plus a stage-level unit test.

**`state/models.py` extensions** — add SQLAlchemy 2.0 declarative
models for the five remaining tables Report reads, and extend `Finding`
with the dedupe-managed and trace-managed columns the v0.3 model
intentionally omitted. All adds; no migrations. The fields newly mapped
on `Finding`:

- `primary_finding_id: Mapped[str | None]`
- `dedupe_cluster_id: Mapped[str | None]`
- `dedupe_role: Mapped[str | None]`
- `root_cause_summary: Mapped[str | None]`
- `reachable: Mapped[str | None]`
- `validated_at: Mapped[str | None]`
- `superseded_at: Mapped[str | None]`

New models follow the same conventions used by v0.2/v0.3 (string ULID
PKs, `mapped_column` declarations, no relationships in v1.0 — Report
queries each table with explicit filters and joins in Python).

**`orchestrator.run_scan` extension** — after the existing
`runs.status='completed'` finalization, before printing the summary:

```python
if cfg.auto_render and recon_result.outcome == "completed":
    try:
        report_result = stages.report.render(
            run_id=run_id,
            session_factory=session_factory,
            output_dir=cfg.report_output_dir,
            formats=cfg.report_formats,
        )
    except FlosswingError as exc:
        # Report failures are surfaced but do not retro-mark the run
        # as errored. The scan itself ran fine; only the render failed.
        return RunResult(exit_code=1, summary=_render_failure_msg(exc))
```

Run-finalization rules:

- Recon errored / 0 tasks queued → `runs.status='errored'`, exit 1, no
  Report attempted (nothing meaningful to render).
- Hunt all-tasks-failed → `runs.status='errored'`, exit 1, no Report
  attempted.
- Scan succeeded and `auto_render=True` and render succeeded →
  `runs.status='completed'`, exit 0, summary points operator at
  `output_dir`.
- Scan succeeded and `auto_render=True` and render FAILED →
  `runs.status='completed'` (do not retro-mark), exit 1, summary tells
  operator to run `flosswing report <run_id>` manually.
- Scan succeeded and `auto_render=False` (operator passed `--no-report`)
  → `runs.status='completed'`, exit 0, summary mentions
  `flosswing report <run_id>` as the next step.

**`cli.py` extensions**:

- `scan` gains `--no-report` (boolean, default off, i.e. auto-render on
  by default) and `--format` (multi-value, default `md,json`).
- `report` is implemented:

```python
@main.command()
@click.argument("run_id")
@click.option("--format", "formats", default="md,json",
              help="Comma-separated formats. Accepts md, json, sarif.")
@click.option("--output-dir", default=None,
              type=click.Path(file_okay=False, dir_okay=True),
              help="Override output dir (default "
                   "~/.flosswing/runs/<run_id>/output/).")
def report(run_id: str, formats: str, output_dir: str | None) -> None:
    """Render the report for an existing RUN_ID."""
    ...
```

Behaviour: re-runs `stages.report.render(...)` against the existing
state DB. If `run_id` is unknown, exits 2 with a clean message. Does
not modify any state-DB row.

### JSON schema — `ReportV1`

Operator-facing Pydantic model. **Not subject to
`docs/tool-contracts.md`'s frozen-API rule** (that document explicitly
calls out Report as "not in this contract"). The schema is treated as a
stable artifact and carries an explicit `schema_version: "1.0"`
literal; breaking changes increment to `"2.0"` and ship under a new
milestone. Non-breaking additions (new optional fields) are allowed
within `1.x`.

Sketch (full Pydantic definitions in the implementation):

```python
class ReportV1(BaseModel):
    schema_version: Literal["1.0"] = "1.0"
    run: ReportRun
    summary: ReportSummary
    findings: list[ReportFinding]
    dedupe_clusters: list[ReportDedupeCluster]
    disclaimer: str  # "A FlossWing scan does not mean the target is secure."
    rendered_at: str  # ISO-8601 UTC

class ReportFinding(BaseModel):
    id: str
    attack_class: str
    file: str
    function: str | None
    line_start: int
    line_end: int
    severity: Literal["critical", "high", "medium", "low", "info"]
    confidence: Literal["confirmed", "likely", "speculative"]
    status: Literal["pending_validation", "confirmed", "rejected",
                    "uncertain", "superseded"]
    reachability: Literal["reachable", "unreachable", "uncertain"] | None
    title: str
    description: str            # markdown, verbatim from DB
    poc_code: str | None        # source, verbatim
    poc_result: dict | None     # parsed JSON, verbatim
    suggested_fix: str | None
    validation: ReportValidation | None
    trace: ReportTrace | None
    dedupe_role: Literal["primary", "variant", "duplicate"] | None
    primary_finding_id: str | None
```

### Markdown rendering

Pure-string assembly. Section order:

1. Run header — run_id, target repo path, started/finished timestamps,
   tokens, cost.
2. **Disclaimer block** — verbatim "A FlossWing scan does not mean the
   target is secure." plus a short paragraph from ARCHITECTURE.md §
   Threat model summary point 5 reworded for human readers.
3. Summary — counts by severity, confidence, status, reachability.
4. Findings table — sortable groups. v1.0 groups primarily by
   **severity** descending (critical → info), then within each severity
   group by **reachability** (reachable → uncertain → unreachable →
   unknown). See Open Questions for the reverse-order debate.
5. Per-finding sections — one Markdown section per finding, containing
   title, location (`file:line_start-line_end`), attack class,
   severity/confidence/status badges, description, optional PoC code
   block, optional validation summary, optional trace summary, optional
   suggested fix.
6. Dedupe clusters appendix — one section per cluster with the primary
   finding link and the list of variants/duplicates.
7. Agent-session audit table — per-stage token usage, cost, outcomes,
   refusals (refusal text scrubbed).

Untrusted-input handling: see § Security considerations. Critically,
attacker-supplied strings are never rendered as anything but
fenced code or quoted markdown blockquote. They are not interpolated
into headings, table cells without escaping, or HTML.

### Per-finding directories

For each finding with `status='confirmed'`:

```
findings/<finding_id>/
  finding.json           # ReportFinding projection
  poc/                   # always present; contents depend on poc_code
    <basename>.<ext>     # if poc_code is a single source string,
                         # written as e.g. poc.py / poc.c (extension
                         # guessed from `attack_class.language_scope`
                         # and the finding's file extension; default
                         # poc.txt if ambiguous)
    output.txt           # if poc_result_json contains stdout/stderr,
                         # written verbatim; absent otherwise
  trace.json             # if a traces row exists for the finding
  suggested_fix.md       # if suggested_fix is non-null
```

`poc_code` is **never executed** by Report. It is written to disk as a
file and never invoked. The operator runs it manually if they choose
to, in their own environment, after their own review.

**Findings with `status` other than `confirmed` are intentionally
excluded** from `findings/<id>/` per ARCHITECTURE.md § Stage 8. They
appear in `report.md` (in the body) and in `report.json` (in the
`findings` list, with their actual status), but do not get their own
artifact directory. This is a conscious operator-attention budget
choice: per-finding directories are the "you should act on this" tier.

## Graceful degradation — Report running before Validate / Trace / Dedupe land

Report ships before the v0.4–v0.7 plumbing slices for Validate,
Gapfill, Dedupe, and Trace. Until those land, the state DB will be
populated only with `findings` rows from Hunt — every finding will
have `status='pending_validation'`, no `validations` rows, no `traces`
rows, no `dedupe_clusters` rows. The renderer must handle this without
crashing or emitting confusing language. Specifically:

- `report.md` renders the findings list with explicit status badges; a
  pending-validation finding renders as "Pending validation (v1.0
  Report ran before Validate stage landed)".
- `report.json` `findings[].validation` and `findings[].trace` are
  `None` for unvalidated/untraced findings; `dedupe_clusters` is `[]`.
- `findings/<id>/` directories are emitted only for CONFIRMED findings;
  pre-Validate runs produce **zero** per-finding directories. The
  integration test asserts an empty-but-present `findings/` dir for the
  v0.3 smoke corpus.
- The render disclaimer block notes that absent validation/trace data
  means findings are **less reliable**, not more — exactly the opposite
  of what an inattentive operator might infer from "fewer warnings."

When v0.4 (Validate) lands, the same renderer picks up the new rows
without code changes; this is the point of doing Report as a
deterministic projection over a fixed schema.

## Data flow — one `flosswing scan <path>` run with v1.0

```
1. CLI parses args            → cli.scan(path, model?, recon_budget?,
                                          hunt_budget?, no_report?,
                                          format?)
2. config.resolve()           → unchanged; adds report_formats +
                                report_output_dir + auto_render
3. orchestrator.run_scan(repo, cfg)
   a. ensure state DB exists                            [v0.2]
   b. generate run_id, create scratch dir, insert runs row
   c. await stages.recon.run(...)                       [v0.2]
   d. if recon ok → await stages.hunt.run(...)          [v0.3]
   e. orchestrator finalize:
        - finalize runs.status based on stage outcomes  [v0.3]
        - if status=='completed' AND auto_render:
            try:
                report_result = stages.report.render(
                    run_id=run_id,
                    session_factory=session_factory,
                    output_dir=cfg.report_output_dir,
                    formats=cfg.report_formats,
                )
            except FlosswingError as e:
                summary += "report rendering failed: " + scrub(str(e))
                exit_code = 1
        - print summary (now includes output_dir if rendered)
4. Process exits

`flosswing report <run_id>` alt path:
1. config.resolve(no repo path)                  → loads state DB only
2. stages.report.render(run_id=..., session_factory=..., ...)
3. Print files-written summary
4. Exit 0 (or 2 if run_id is unknown / 1 on render error)
```

Single-writer assumption from the schema still holds — only the
orchestrator process writes to the state DB. Report **never** writes to
the state DB; its only filesystem writes are inside `output_dir`.

## Determinism

`stages.report.render()` is deterministic given a fixed state DB and a
fixed `rendered_at`. To make output byte-identical across runs (which
the success criteria above promise modulo a single timestamp field), we
follow these rules:

- All SELECT queries carry explicit `ORDER BY` clauses on stable
  columns (typically the ULID `id`, which is monotonic by creation
  time).
- Markdown grouping order (severity, then reachability) uses an
  explicit dict-based ordering — no `sorted()` against enum string
  ordering, which would re-order alphabetically.
- `rendered_at` is the **only** non-deterministic field, and is
  isolated in one place (`ReportV1.rendered_at`) so tests can diff
  output by stubbing it. Tests use a fixed `rendered_at` injected via
  a `_now: Callable[[], str]` keyword on `render()`; default is
  `datetime.now(timezone.utc).isoformat()`.
- `report.json` is pretty-printed with `indent=2` for diffability; the
  JSON object key order matches Pydantic field order (Pydantic
  guarantees this for `.model_dump_json()`).

## Error handling

Report inherits all v0.2/v0.3 categories. New ones:

| Condition | Where caught | What's stored | CLI behavior |
|---|---|---|---|
| `run_id` unknown to `flosswing report` | `stages.report._load_run_data` raises `RunNotFoundError` | nothing | exit 2; "no run with id <…> in state.db" |
| Output dir cannot be created / written | `stages.report.render` wraps OSError | nothing | exit 1; print scrubbed error |
| State DB unreadable / migration mismatch | `state.session.engine()` (v0.2 behavior) | nothing | exit 1; existing message |
| Pydantic projection fails (schema/state mismatch) | `stages.report._build_report_model` | nothing | exit 1; "report projection failed: <code>" — code is the Pydantic validation error's `loc` path, **not** the raw value, to avoid leaking finding contents into stderr |
| Per-finding-dir write partial failure | `_write_per_finding_dirs` | nothing | exit 1; abort the render, leave whatever has been written in place; print "partial render: see <output_dir>" |
| Auto-render failure during scan | `orchestrator.run_scan` | run row stays `completed` | exit 1; tell operator to re-run `flosswing report <run_id>` |

`errors.scrub()` runs on every string going to stdout/stderr. The
`description`, `poc_code`, and `suggested_fix` fields are NOT scrubbed
when written to `report.md` / `report.json` / per-finding artifacts —
that would corrupt the content. They ARE rendered with the
prompt-injection-safe rules in § Security considerations.

## Security considerations

The target repo is untrusted. Findings produced by Hunt may contain
strings sourced directly from repo contents (function names, snippets,
PoC code, the Hunter's description text — which itself may quote the
repo). Same untrusted-input rules apply to anything Report renders from
these fields:

- **`description`, `suggested_fix`** — written verbatim into a markdown
  document. They MAY contain markdown syntax. They MUST NOT be passed
  through any HTML renderer, templating engine, or shell-expansion
  step. Hand-rolled string assembly means there is no template engine
  to abuse; we just concatenate. We do **not** strip or sanitize
  markdown — this is documentation for an operator who is reading it
  as markdown, not rendered HTML. A README that says "execute `rm -rf
  /`" in a code fence is fine; it's just text.
- **`poc_code`** — written verbatim into `report.md` as a fenced code
  block with a language hint guessed from the finding's file
  extension. **Never executed.** The fence guarantees the markdown
  renderer (whatever the operator uses) treats the content as code,
  not as markdown to interpret.
- **`title`, `function`, `file`** — rendered into headings and tables.
  These need escaping to prevent injection of markdown control
  characters that break table rendering (a stray `|` in `function`
  closes the cell). Escape: replace `|` with `\|`, replace any literal
  newline with a space, truncate to a max of 200 chars with `…` suffix.
  Implemented in a single `_escape_table_cell` helper.
- **`findings/<id>/poc/` directory contents** — written byte-for-byte
  from `poc_code` to a file. Filenames are derived from
  `<finding_id>` (server-side ULID, not attacker-controlled) plus a
  language-guessed extension. The attacker controls the **contents** of
  the file, not the path; the path cannot escape `output_dir`.
- **Refusal text in the agent-session audit table** — already scrubbed
  by `errors.scrub()` at the point of being stored in
  `agent_sessions.refusal_text` (v0.2 behavior). Report reads the
  already-scrubbed value and writes it verbatim; no double scrub.

Worth restating explicitly because of the temptation to "make the
report look nicer": **do not introduce any HTML-rendering step, any
markdown→HTML conversion, or any templating engine.** Markdown stays
markdown. If a v1.1 HTML output ever happens, it gets its own
sandboxing design.

## SARIF stance

ARCHITECTURE.md § v1 scope says SARIF "may land in v1.1 if cheap."
v1.0 ships **a CLI shim only**: the `--format sarif` flag parses, the
formats set contains `sarif`, and `render()` emits a placeholder file
`report.sarif` containing exactly:

```
{"$comment": "sarif output is not yet implemented; tracked in v1.1"}
```

— valid JSON, parseable by any consumer, with a clear signal. The
flag exists so existing CI configs that ask for SARIF don't error;
they just get a placeholder until v1.1.

When SARIF lands in v1.1, the recommendation is **hand-roll the JSON**
rather than add a `sarif-om` or similar dependency. SARIF 2.1.0 is
~30 well-defined fields for a code-scanning use case; the per-finding
projection in v1.0 already covers most of them; Pydantic models for
the SARIF shapes are a couple hundred lines. Operator decision pending
— see Open Questions.

## Testing strategy

### Unit tests (in normal CI)

- `tests/unit/test_stages_report_empty.py` — render a run with one
  `runs` row, one `recon_artifacts` row, ≥1 `hunt_tasks` row, zero
  `findings`. Asserts `report.md` and `report.json` are written, no
  `findings/<id>/` dirs exist, disclaimer is present, agent-session
  audit table is present.
- `tests/unit/test_stages_report_with_findings.py` — render a run with
  multiple findings spanning severities, statuses, reachabilities.
  Assert grouping order, per-finding section presence, that
  CONFIRMED findings get a dir and others don't, that
  `findings/<id>/finding.json` round-trips through `ReportFinding`.
- `tests/unit/test_stages_report_untrusted_input.py` — fabricate a
  finding with `description` and `title` containing markdown
  metacharacters (`|`, `[`, backticks, table-row HTML), `poc_code`
  with a fake shell command. Assert the rendered markdown has table
  cells escaped, the code block is fenced and language-hinted, the
  file written to `poc/` matches `poc_code` byte-for-byte, and no
  `<script>` or unescaped HTML leaks into `report.md` (assert via a
  blocklist scan of the rendered string).
- `tests/unit/test_stages_report_determinism.py` — render the same DB
  twice with a stubbed `_now`; assert `report.md` and `report.json`
  are byte-identical.
- `tests/unit/test_stages_report_pending_validation.py` — render a run
  where every finding is `pending_validation` (i.e. Validate hasn't
  landed yet). Assert no `findings/<id>/` dirs; assert
  `report.md` notes "Validate stage has not yet rendered verdicts"
  language.
- `tests/unit/test_cli_report.py` — `flosswing report <run_id>`:
  unknown run id → exit 2; known run id → re-render produces same
  files; `--format sarif` produces the placeholder.

Per `CLAUDE.md § Testing`: no SDK mock needed (no agent involved).
All tests use an in-memory SQLite engine with rows inserted via the
existing models, no fixtures-on-disk.

### Integration test (gated, NOT in CI)

Extend `tests/integration/test_recon_smoke.py` once more. After the
existing v0.3 Hunt-side assertions, add:

- `output_dir` exists at `~/.flosswing/runs/<run_id>/output/`.
- `report.md` exists, contains the disclaimer substring, references
  the `command_injection` finding from the v0.3 smoke corpus.
- `report.json` parses as `ReportV1` and `len(report.findings) >= 1`.
- `findings/` exists; in v1.0-only mode (no Validate yet) it is empty,
  asserted as such. Once Validate lands, this assertion flips.

### Eval

`flosswing eval` remains a stub. Eval against `tests/corpus/` is its
own milestone.

### CI changes

None for v1.0. Existing pipeline (`ruff check`, `mypy --strict
flosswing`, `alembic upgrade/downgrade round-trip`, `pytest
tests/unit`) covers everything.

## Open questions / decisions pending

1. **Auto-render vs. explicit subcommand only.** Current spec
   auto-renders at end of every successful scan and ALSO exposes
   `flosswing report <run_id>` for re-rendering. The alternative is
   to make `flosswing report <run_id>` the only way to render, never
   auto-running. Auto-render is the better default because the
   ARCHITECTURE.md § Operating model description literally says "runs
   the pipeline, writes findings to its own state directory, **generates
   a report**" — but it's an operator preference call. Default
   `auto_render=True` proposed; `--no-report` opts out.
2. **SARIF in v1.0 vs v1.1.** Current spec ships only a placeholder in
   v1.0 and reserves real SARIF for v1.1. The alternative is to push
   SARIF into v1.0 (hand-rolled, no new dep) since it's the format
   most enterprises actually consume. Recommendation: defer. Real
   SARIF needs a careful mapping of severity / confidence / status
   onto SARIF's `level` and `properties.tags`, and that's its own
   design — too much for v1.0.
3. **Finding ordering — severity first or reachability first?** v1.0
   spec groups by severity descending, then reachability. The
   counter-proposal is reachability first (reachable findings get
   first attention, severity is a tiebreaker within reachable). The
   Cloudflare prior art groups by severity. We follow that for v1.0;
   revisit once we have operator usage data.
4. **How to represent UNCERTAIN findings.** `Finding.status` can be
   `uncertain` (set by Validate when it can't decide). v1.0 spec puts
   them in `report.md` alongside CONFIRMED findings in the main
   sections, marked with an explicit "uncertain" badge, but does
   **not** give them their own `findings/<id>/` directory. The
   alternative is to put them in their own section ("Needs operator
   review") between CONFIRMED and lower-confidence findings. Decision
   pending; current spec is the simpler one.
5. **`report.json` schema versioning.** v1.0 ships `schema_version:
   "1.0"` as a literal field. Operator decision pending whether to
   commit to that version-ing convention now (mirroring
   `tool-contracts.md`'s versioning policy) or treat the JSON shape
   as freely evolving until the operator says otherwise. The spec's
   working answer is "stable, non-breaking adds within 1.x, breaking
   changes bump to 2.0", but this is the only spec where the schema
   is operator-facing rather than agent-facing and the call could go
   either way.
6. **Output directory location.** Spec follows ARCHITECTURE.md § Stage
   8 verbatim: `~/.flosswing/runs/<run_id>/output/`. There is a
   long-standing contradiction in ARCHITECTURE.md between "state lives
   in `<target>/.flosswing/`" (§ Operating model) and "state lives in
   `~/.flosswing/`" (§ State store), flagged in the v0.2 spec and
   still unresolved. v1.0 picks the home-dir path consistently — same
   place v0.2 puts the state DB — but the inconsistency in
   ARCHITECTURE.md should be resolved before v1.0 lands.
7. **Single file vs. package layout.** `stages/report.py` is one file
   in v1.0. If markdown rendering plus per-finding-dir writing plus
   the JSON projection plus SARIF (when it lands) push the file past
   ~600 lines, lift to `flosswing/report/` with submodules
   (`markdown.py`, `json_.py`, `sarif.py`, `findings_dirs.py`,
   `_load.py`). v1.0 explicitly does **not** preemptively split — the
   v0.3 attack-class fragment loader's "one consumer, don't abstract
   yet" reasoning applies symmetrically here.

## Definition of "done" for v1.0

When the success criteria above all pass, plus:

- `flosswing scan tests/corpus/v02_smoke/` runs Recon → Hunt → Report
  end-to-end, producing a `~/.flosswing/runs/<run_id>/output/`
  directory containing `report.md`, `report.json`, and a present-but-
  empty `findings/` directory (because Validate hasn't landed).
- `flosswing report <run_id>` against that run re-renders byte-
  identical output (modulo `rendered_at`).
- The ARCHITECTURE.md amendments above land in the same commit as
  `stages/report.py` (or the immediately following commit, per the
  v0.2/v0.3 pattern).
- `pytest tests/unit` runs at least six new test modules (the six
  `test_stages_report_*` and `test_cli_report.py`) and all pass.
- `report.json` validates against `ReportV1.model_validate_json(...)`
  for every run produced by the existing v0.3 smoke corpus and at
  least one fabricated multi-finding fixture.
