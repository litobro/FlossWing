## Context

v0.2 (Recon plumbing) shipped the agent runtime end-to-end: `flosswing
scan <path>` spawns the `claude` CLI, runs Recon, records one
`recon_artifacts` row and ≥1 `hunt_tasks`, and audits the session.
v0.3 (Hunt plumbing) extended the walk so the orchestrator dequeues
pending hunt tasks and lets a Hunter agent record `findings` against
`tests/corpus/v02_smoke/`. Neither milestone touched execution: the
Hunt prompt forbids `confidence='confirmed'` precisely because
`compile_and_run` and the sandbox behind it are not yet wired.

v0.4 is the **smallest vertical slice that proves the sandbox runtime
works end-to-end** — orthogonal to v0.3 in concern (Hunt vs. execution)
but built on the same orchestrator and tool-registry plumbing v0.2
established. It implements just enough of the sandbox to:

- detect whether Docker or Firejail (or neither) is available,
- materialize the per-attack-class network policy,
- write a caller's `files` to a per-invocation scratch directory,
- launch one container/jail per `compile_and_run` invocation under the
  ARCH-mandated constraints (`--network=none`, read-only root, tmpfs
  `/scratch`, `/repo` read-only, `--memory=2g`, `--cpus=2`,
  `--pids-limit=256`, `--cap-drop=ALL`, wall-clock timeout,
  stdout/stderr size caps),
- return a structured `CompileAndRunOutput` matching the frozen
  contract, including `sandbox_backend` and `scratch_path`,
- return `sandbox_unavailable` cleanly when neither backend is
  installed, so callers degrade gracefully.

Milestone numbering is provisional; the operator can reorder this
relative to v0.5 symbol-index, v0.6 Validate, etc. v0.4 was chosen
because every subsequent stage that wants real PoC execution (Hunt
with `confidence='confirmed'`, Validate's disproof attempts) is
blocked on the sandbox layer, and because sandboxing has the longest
implementation tail of any v1 component — better to land it now on a
deliberate corpus PoC than under pressure from a stage that needs it.

Hunt and Validate **do not gain `compile_and_run` in v0.4**. Wiring
the tool into their scopes is a follow-on milestone that consumes
this layer. v0.4 verifies the plumbing through a direct harness
invocation — a deliberate `command_injection` PoC against `v02_smoke`
that compiles and runs inside the sandbox, demonstrates
`--network=none` is enforced, and returns the structured `ExecResult`
with all caps observed. That keeps the change surface narrow and the
Hunt prompt unchanged.

Anything hard or expensive (libFuzzer/AFL integration, seccomp
profiles, AppArmor / SELinux interplay, container-image SBOM tracking,
network-namespace loopback fixtures for SSRF) is intentionally cut.

## Success criteria

A v0.4 implementation is done when **all** of the following are true:

1. `flosswing.sandbox.docker.DockerSandbox` and
   `flosswing.sandbox.firejail.FirejailSandbox` both implement the
   `flosswing.sandbox.base.Sandbox` Protocol; either can be selected
   via `flosswing.sandbox.select_backend()`.
2. `select_backend()` returns the Docker backend if `docker info`
   succeeds, otherwise Firejail if `firejail --version` succeeds,
   otherwise raises `SandboxUnavailableError` (already defined in
   `errors.py` per v0.2) with `code='sandbox_unavailable'` and
   `retryable=False` — matching the frozen tool-contracts error.
3. A deliberate test PoC compiles and runs to completion inside the
   sandbox, returning a populated `CompileAndRunOutput` with
   `run.exit_code == 0`, `run.duration_ms > 0`,
   `run.network_used == False`, `run.sandbox_backend ∈ {"docker",
   "firejail"}`, and `scratch_path` pointing to the host-side
   directory the source files were materialized in (so the future
   `record_finding` caller can reference it).
4. A network-attempt PoC (Python opens TCP to `1.1.1.1:80`) returns
   `run.exit_code != 0`, `run.network_used == True`, and `run.stderr`
   containing the OS-level "Network is unreachable" / `EPERM` /
   `getaddrinfo` failure. Network observation is recorded even when
   blocked.
5. A timeout PoC (Python `time.sleep(120)` with `timeout_seconds=2`)
   returns `run.timed_out == True`, `run.exit_code == -1`,
   `run.signal == "SIGKILL"`, overall wall-clock under ~5 seconds.
6. A resource-cap PoC (allocate 3 GB) returns `run.oom_killed ==
   True`, `run.exit_code == -1`, `run.signal == "SIGKILL"`.
7. `ruff check .`, `mypy --strict flosswing`, and `pytest
   tests/unit` continue to pass. CI gains no new external Python
   dependencies — `docker` is already in the approved list.
8. Existing v0.2/v0.3 unit and integration tests still pass; v0.4
   does not modify Recon or Hunt behaviour.
9. `alembic upgrade head` round-trip continues to pass — **schema is
   unchanged in v0.4**.

## Scope

### In scope for v0.4

- New modules: `flosswing/sandbox/{__init__,base,docker,firejail,policy}.py`,
  `flosswing/sandbox/images/<lang>.Dockerfile` (one per language in
  the frozen `CompileAndRunInput.language` literal: `c`, `cpp`,
  `rust`, `go`, `python`, `javascript`, `typescript`, `java`).
- `flosswing/tools/execution.py` — wraps the sandbox in the frozen
  `compile_and_run` contract. **Implementation complete**; what's
  deferred is registering it into Hunt or Validate scopes.
- Per-attack-class network policy lookup via a new `network_default`
  and `network_permitted` field on `AttackClassMeta`. Default
  `False`/`False` for every v1 class except `ssrf`
  (`permitted=True`, but the loopback fixture is itself deferred).
- Filesystem layout for per-invocation scratch under
  `~/.flosswing/runs/<run_id>/sandbox/<invocation_id>/` on the host,
  bind-mounted into the container at `/scratch/src`. `scratch_path`
  in the returned `CompileAndRunOutput` is the **host** path so a
  future `record_finding` (running in the harness, not the
  container) can reference it.
- Wall-clock timeout enforced **outside** the container by the
  driver (Docker SDK kill-on-timeout, or `subprocess.run(timeout=…)`
  for Firejail). `timeout(1)`-inside-container from ARCHITECTURE.md
  is a belt-and-suspenders second-layer guard inside the language
  Dockerfiles; either layer killing suffices.
- Stdout/stderr captured to bounded buffers at 10 MB each, with
  `stdout_truncated` / `stderr_truncated` flags.
- Backend selection cached for the process lifetime so a long Hunt
  run doesn't re-detect on every call.
- Unit tests covering: backend selection, policy lookup, scratch
  layout, Docker-SDK kwargs (this is the test that catches
  accidental cap relaxation), env-allowlist filter.
- One integration test gated by **both** `FLOSSWING_INTEGRATION=1`
  and `FLOSSWING_SANDBOX_INTEGRATION=1` (so existing Recon/Hunt
  integration runs without Docker installed don't fail). Runs the
  four PoCs from § Success criteria 3-6.
- One paragraph appended to `ARCHITECTURE.md § Sandbox` clarifying
  v0.4-specific scope (proposed text below).

### Out of scope for v0.4 (deferred)

- **Wiring `compile_and_run` into Hunt or Validate scopes.** Tool
  lives in `flosswing/tools/execution.py` but no stage registers it
  in v0.4. Hunt continues with the v0.3 tool set; the Hunt
  `confidence='confirmed'` prohibition stays.
- libFuzzer / AFL++ integration. ARCHITECTURE.md says "FlossWing
  may invoke fuzzers as tools inside the sandbox" — that's a future
  tool surface (e.g. `run_fuzzer`), not v0.4.
- Per-language Dockerfile *content* beyond stubs. Each Dockerfile
  in v0.4 pins a base image and installs the build toolchain for
  the language's default `build_command`. No package caching, no
  pre-compiled dependencies, no language-version matrix.
- Hardening beyond ARCH-mandated constraints. No seccomp profile,
  no AppArmor, no SELinux labels, no user-namespace remapping
  beyond Docker's defaults. Flagged as future work.
- Image build pipeline / image registry / pre-pushed images. v0.4
  builds lazily on first use of a language.
- SSRF loopback HTTP fixture. Policy plumbing is in v0.4; the
  actual fixture ships with the SSRF attack-class prompt.
- Per-run cleanup of `~/.flosswing/runs/<run_id>/sandbox/`. v0.4
  leaves scratch on disk so the operator can inspect. A
  `flosswing prune` subcommand is v2.
- Schema-vs-models CI drift check. Same stance as v0.2/v0.3.

### Explicit non-goals (per `ARCHITECTURE.md`)

Unchanged from v0.2/v0.3: no daemon, no SaaS, no auto-disclosure,
no auto-patching, no telemetry, no cross-repo, no fuzzing framework,
no non-Anthropic providers, no write access to the target repo. The
sandbox layer is the single highest-risk surface (model writes
attacker-controlled code; harness executes it), so every
ARCH-mandated constraint is implemented verbatim — not paraphrased,
not relaxed.

## Architecture

### Module layout

```
flosswing/
  sandbox/
    __init__.py            (NEW; exports select_backend, Sandbox, ExecResult)
    base.py                (NEW; Sandbox Protocol + shared dataclasses)
    docker.py              (NEW; DockerSandbox)
    firejail.py            (NEW; FirejailSandbox)
    policy.py              (NEW; per-attack-class network policy)
    images/
      c.Dockerfile         (NEW; stub: debian-slim + build-essential)
      cpp.Dockerfile       (NEW; stub)
      rust.Dockerfile      (NEW; stub: rustup + cargo)
      go.Dockerfile        (NEW; stub: golang)
      python.Dockerfile    (NEW; stub: python:3.11-slim)
      javascript.Dockerfile (NEW; stub: node)
      typescript.Dockerfile (NEW; stub: node + typescript)
      java.Dockerfile      (NEW; stub: temurin)

  tools/
    execution.py           (NEW; compile_and_run wrapper)

  attack_classes.py        (existing — extend AttackClassMeta)
  errors.py                (existing — SandboxUnavailableError already in v0.2)
```

No new top-level Python dependencies. `docker` was added in
`CLAUDE.md § Dependency policy` precisely for this milestone.
Firejail is a system binary (no Python package).

### Component responsibilities

**`sandbox/base.py`** — the Protocol every backend implements:

```python
class SandboxInvocation(BaseModel):
    invocation_id: str         # ULID; matches scratch dir name
    run_id: str
    attack_class: str
    language: str
    files: list[SourceFile]
    build_command: str | None
    run_command: str
    stdin: str | None
    args: list[str]
    env: dict[str, str]        # caller-provided, allowlist-filtered
    timeout_seconds: int       # validated 1..300
    network: bool              # validated against policy

class Sandbox(Protocol):
    backend_name: str          # "docker" | "firejail"
    async def is_available(self) -> bool: ...
    async def execute(
        self,
        invocation: SandboxInvocation,
        repo_root: Path,
    ) -> CompileAndRunOutput: ...
```

The Protocol intentionally does **not** include `cleanup` in v0.4 —
scratch dirs persist until the operator removes them.

**`sandbox/docker.py`** — primary backend. Per invocation:

1. Verify the language image `flosswing-sandbox-<language>:v0.4`
   exists locally; if not, build from `images/<language>.Dockerfile`
   via the Docker SDK. Build failures raise
   `SandboxImageBuildError` → `sandbox_unavailable` at the tool layer.
2. Create host scratch
   `~/.flosswing/runs/<run_id>/sandbox/<invocation_id>/` and write
   each `SourceFile` into `src/`. `relative_path` segments
   containing `..` raise `PathEscapesScratchError`, mapped to the
   v0.2 `input_validation_failed` error code (per design decision
   #6) to avoid touching the frozen contract.
3. Launch the container with these flags, all set explicitly via
   the Docker SDK `containers.run(...)` call — never via a shell:
   - `image=flosswing-sandbox-<language>:v0.4`
   - `network_mode="none"` (unless policy permits and caller asked)
   - `read_only=True`
   - `tmpfs={"/scratch/work": "size=128m"}` (per design decision #3)
   - `volumes={host_src: {"bind": "/scratch/src", "mode": "rw"},
                repo_root: {"bind": "/repo",       "mode": "ro"}}`
   - `mem_limit="2g"`, `nano_cpus=2_000_000_000` (== 2 CPUs in
     Docker's billionths-of-a-CPU unit), `pids_limit=256`
   - `cap_drop=["ALL"]`, no `cap_add`
   - `user="65534:65534"` (nobody:nogroup)
   - `working_dir="/scratch/work"`
   - `environment=<filtered env>`
   - `entrypoint=["/bin/sh", "-lc"]`, `command=<build+run script>`
     where the script copies `/scratch/src/*` into
     `/scratch/work/`, runs `build_command` (if any), then
     `run_command` with `args`
   - `detach=False`, wrapped in `asyncio.wait_for(timeout=…)`
4. On timeout: `container.kill('SIGKILL')`, set `timed_out=True`,
   `exit_code=-1`, `signal='SIGKILL'`.
5. On OOM: detected via `container.attrs["State"]["OOMKilled"]`.
6. Network attempts inferred best-effort from stderr patterns
   (`EPERM`, `ENETUNREACH`, `getaddrinfo`, `socket.gaierror`).
   Precise observability is an open question.
7. Logs captured via `container.logs(stdout=True, stderr=True,
   stream=False)`, truncated at 10 MB each.

**`sandbox/firejail.py`** — fallback backend. Translates the same
invocation into a `firejail` command line:

- `--net=none` (or controlled namespace if policy permits — same
  deferred branch as Docker)
- `--read-only=/` plus targeted paths firejail needs explicitly
- `--private`, `--private-tmp`
- `--bind <host-scratch>:/scratch/src` via the non-root path
  (`--whitelist`); v0.4 does not require sudo
- `--rlimit-as=2000000000`, `--rlimit-nproc=256`, `--cpu=0,1`,
  `--caps.drop=all`
- `--timeout=00:00:NN` (firejail's enforcement) plus
  `subprocess.run(timeout=…)` as the outer guard
- `--quiet`, `--noprofile` (skip language-aware profiles that can
  subtly relax things)

Firejail does **not** provide per-language filesystem images. When
Firejail is the active backend, the host must have the language
toolchain installed. Deliberate trade-off — Firejail is a fallback
for environments where Docker is unavailable. Documented in the
ARCH amendment.

**`sandbox/policy.py`** — per-attack-class network policy. Extends
`AttackClassMeta`:

```python
@dataclass(frozen=True)
class AttackClassMeta:
    name: str
    language_scope: str              # existing
    network_default: bool = False    # NEW in v0.4
    network_permitted: bool = False  # NEW: whether opt-in is allowed
```

`network_default` is the policy default; `network_permitted` gates
the caller's explicit `network=True` request. In v0.4 only `ssrf`
sets `network_permitted=True`, and since the SSRF fixture is
deferred, no class effectively permits network in v0.4.

**`sandbox/__init__.py`** — `select_backend(preferred: str | None =
None) -> Sandbox`:

- `preferred="docker"`: returns Docker if available, else
  `SandboxUnavailableError` (does NOT fall back to Firejail).
- `preferred="firejail"`: same shape, Firejail only.
- `preferred=None` (auto): Docker if available, else Firejail if
  available, else `SandboxUnavailableError`.

`DockerSandbox.is_available()` runs `docker.from_env().ping()` with
a short timeout. `FirejailSandbox.is_available()` runs `firejail
--version` via subprocess. Both cache the result for the process.

**`tools/execution.py`** — wraps the sandbox in the frozen
`compile_and_run` contract:

- Validate `timeout_seconds ∈ 1..300` → `resource_limit_exceeded`
  otherwise.
- Validate `language` is in the frozen literal (Pydantic already
  enforces; explicit check is defensive) → `language_not_supported`.
- `policy.lookup(attack_class).network_permitted` vs caller's
  `network` → `network_not_permitted` on mismatch.
- Filter `env` against an allowlist (see § Env filtering).
- Generate `invocation_id` (ULID).
- `await backend.execute(invocation, repo_root=ctx.repo_root)`.
- Catch `SandboxUnavailableError` → `sandbox_unavailable`
  (`retryable=False`).
- Return `CompileAndRunOutput` produced by the backend.

The tool wrapper does **not** write to the state DB. Recording the
result is `record_finding`'s job via `poc_result` on
`RecordFindingInput`. Until Hunt gains `compile_and_run`, no DB
writes flow from the sandbox.

### Env filtering

`invocation.env` is passed through after filtering against an
allowlist:

- `LANG`, `LC_ALL`
- `PYTHONUNBUFFERED`
- `RUSTFLAGS`, `CARGO_HOME`
- `GOFLAGS`, `GOPATH`
- `NODE_OPTIONS`

Anything else is dropped silently. Deliberately tight — we
explicitly do **not** forward `ANTHROPIC_*` / `AZURE_*` / `AWS_*` /
`GITHUB_*` regardless of opt-in. The credential rule from
`CLAUDE.md § Hard rules` forbids it.

### Filesystem layout

```
~/.flosswing/runs/<run_id>/sandbox/<invocation_id>/
  meta.json       # invocation_id, attack_class, language,
                  # build_command, run_command, started_at,
                  # image, image_digest, sbom (per design decision #4)
  src/            # caller-provided SourceFiles materialized
  stdout          # captured stdout (pre-truncation)
  stderr          # captured stderr (pre-truncation)
  result.json     # CompileAndRunOutput serialized
```

Per design decision #4: `meta.json` records the base image's
`sha256:…` digest plus a per-language SBOM line captured on first
image build (e.g. `pip freeze` for Python, `cargo --version
--verbose` for Rust). Implementation captures SBOM into the image
layer once, then `docker inspect` reads it back into `meta.json` on
each invocation — no per-invocation cost beyond the read.

The host path is the `scratch_path` returned in the output.
Persisted across runs; cleaned manually. `meta.json` and
`result.json` are written by the harness; the container only writes
to `/scratch/work/` (tmpfs) and never to the host directly.

## Data flow — one `compile_and_run` invocation

```
1. Agent emits compile_and_run with CompileAndRunInput
   (Hunt/Validate, once a follow-on milestone wires the tool).
2. Tool registry shim → tools.execution.compile_and_run:
   a. Validate timeout, language, env allowlist.
   b. policy.lookup(attack_class) → network gate.
   c. Generate invocation_id (ULID).
   d. backend = select_backend(cfg.sandbox_preferred)  [cached]
   e. await backend.execute(SandboxInvocation(...), repo_root)
3. DockerSandbox.execute:
   a. ensure_image(language); build from Dockerfile if missing.
   b. mkdir <scratch>/src; write meta.json; materialize SourceFiles.
   c. docker.containers.run(
        image=…, network_mode='none', read_only=True,
        tmpfs={'/scratch/work':'size=128m'},
        volumes={<scratch>/src→/scratch/src (rw),
                 <repo>→/repo (ro)},
        mem_limit='2g', nano_cpus=2_000_000_000, pids_limit=256,
        cap_drop=['ALL'], user='65534:65534',
        working_dir='/scratch/work',
        environment=<filtered>,
        entrypoint=['/bin/sh','-lc'],
        command=<build+run script>,
        detach=False,
      ) wrapped in asyncio.wait_for(timeout=…+5).
   d. On timeout: container.kill('SIGKILL').
   e. Inspect attrs for OOMKilled, exit code, signal.
   f. Read logs; truncate to 10MB each; write stdout/stderr.
   g. Infer network_used from stderr (best-effort).
   h. Build ExecResult for build (if any) and run; write
      result.json; return CompileAndRunOutput.
4. Tool wrapper returns {"content":[{"type":"text","text":<json>}]}.
```

Concurrency: the v0.4 backend is async-friendly but doesn't itself
run multiple invocations in parallel. When Hunt parallelism lands,
each Hunter gets its own invocation_id and per-container isolation
handles concurrency naturally. No process-wide locks at the sandbox
layer.

## Error and refusal handling

Refusals don't apply at this layer — there is no agent in v0.4's
sandbox loop. The tool layer either succeeds or returns a structured
error to the eventual caller.

| Condition | Where caught | Error code | retryable |
|---|---|---|---|
| Neither Docker nor Firejail available | `select_backend` | `sandbox_unavailable` | `False` |
| Docker daemon down between detect and run | `DockerSandbox.execute` | `sandbox_unavailable` | `False` |
| Language image build fails | `DockerSandbox.execute` | `sandbox_unavailable` (message includes build-log tail) | `False` |
| `language` not in frozen literal | `tools/execution.py` | `language_not_supported` | `False` |
| `network=True` for class not permitting | `tools/execution.py` | `network_not_permitted` | `False` |
| `timeout_seconds` out of 1..300 | `tools/execution.py` | `resource_limit_exceeded` | `False` |
| `relative_path` escapes scratch | backend | `input_validation_failed` (per design decision #6) | `False` |
| Build step exits non-zero | backend | not an error — `build.exit_code != 0` in result | n/a |
| Run step exits non-zero | backend | not an error — `run.exit_code != 0` in result | n/a |
| Container timed out | backend | not an error — `run.timed_out=True`, `signal='SIGKILL'` | n/a |
| Container OOM killed | backend | not an error — `run.oom_killed=True` | n/a |
| Stdout/stderr >10 MB | backend | not an error — truncation flag set | n/a |

The frozen contract distinguishes hard errors (`is_error: True`
tool errors the agent sees) from PoC-level signal (returned in
`ExecResult` fields). v0.4 preserves that distinction strictly:
anything the agent might reasonably react to by changing its PoC
(build failure, runtime failure, timeout, OOM, truncation) is *not*
a tool error.

`errors.scrub()` continues to run on every string going to logs or
the state DB. Sandbox stdout/stderr is **not** scrubbed before
being returned to the agent — the agent needs raw PoC output to
iterate. Scrubbing kicks in only when those streams are written to
`findings.poc_result_json` (Hunt-side) or to operator output.

## Testing strategy

### Unit tests (in normal CI)

- `tests/unit/test_sandbox_select.py` — backend selection.
  Docker present → `DockerSandbox`. Docker absent, Firejail present
  → `FirejailSandbox`. Both absent → `SandboxUnavailableError` with
  `code='sandbox_unavailable'`, `retryable=False`.
  `preferred="docker"` with Docker absent: raises, does NOT fall
  back.
- `tests/unit/test_sandbox_policy.py` — every v1 class returns
  `network_default=False`; only `ssrf` returns
  `network_permitted=True`; unknown class raises `KeyError`.
- `tests/unit/test_tools_execution.py` — `timeout_seconds=400` /
  `=0` → `resource_limit_exceeded`. Unknown `language` (synthetic
  bypass) → `language_not_supported`. `command_injection` +
  `network=True` → `network_not_permitted`. Mocked backend output
  returned unchanged.
- `tests/unit/test_sandbox_docker_args.py` — Docker SDK kwargs.
  Asserts `network_mode='none'`, `read_only=True`,
  `mem_limit='2g'`, `nano_cpus=2_000_000_000`, `pids_limit=256`,
  `cap_drop=['ALL']`, `user='65534:65534'`. This is the test that
  catches accidental cap relaxation.
- `tests/unit/test_sandbox_env_filter.py` — drops
  `ANTHROPIC_API_KEY`, `AZURE_CLIENT_SECRET`, `AWS_SECRET_*`,
  `GITHUB_TOKEN`; retains `LANG`, `PYTHONUNBUFFERED`, etc.

All unit tests mock the Docker SDK at the `docker.from_env()`
boundary, not at HTTP — same rationale as `CLAUDE.md § Testing`
applies to the agent SDK.

### Integration test (gated, NOT in CI)

`tests/integration/test_sandbox_smoke.py` — gated by both
`FLOSSWING_INTEGRATION=1` and `FLOSSWING_SANDBOX_INTEGRATION=1`.
Runs the four PoCs from § Success criteria 3-6 against the real
Docker daemon (or Firejail if `FLOSSWING_SANDBOX_BACKEND=firejail`
is set): happy path, network blocked, timeout, OOM. Each PoC's
source is materialized via the contract (passed in as `SourceFile`),
not pre-baked into a fixture file — mirrors how a real Hunter would
supply PoC code.

### Eval

Still not yet. `flosswing/eval/` content remains its own milestone.

### CI changes

None. Existing pipeline (`ruff check`, `mypy --strict flosswing`,
`alembic upgrade/downgrade round-trip`, `pytest tests/unit`)

## Design decisions (resolved 2026-06-02)

1. **Base image strategy.** Pin every base image by `sha256:…`
   digest (reproducibility under upstream rotation). Default to
   Debian-slim distros — avoid musl quirks at low size cost. Keep
   8 Dockerfiles, one per language family, even when near-identical
   (lets operators tune per-language without surprising other
   langs). Tags get noted alongside the digest as a hint to the
   operator on update.

2. **Firejail availability detection.** Run `firejail --version`
   and treat exit-0 as available. Caps-drop failures at run time
   surface as a clear `sandbox_unavailable` from the first
   `compile_and_run` invocation. Revisit with a `--debug-caps`
   probe only if real Firejail users report silent caps-drop
   failures in practice.

3. **Scratch sizing.** Tighten `/scratch/work` tmpfs to 128 MB
   per invocation (down from 512 MB). The host-side
   `~/.flosswing/runs/<run_id>/sandbox/` directory stays unbounded
   in v0.4; operator cleans manually or via `flosswing prune` (v2).
   Aggressive guard at the hot path; defers the host-dir quota
   story.

4. **Reproducibility metadata.** Each invocation's `meta.json`
   records the base image's `sha256:…` digest **and** a per-language
   SBOM line captured on first image build (e.g. `pip freeze` for
   Python, `cargo --version --verbose` for Rust). SBOM is baked
   into the image layer once and read back via `docker inspect`
   per invocation — no per-invocation rebuild cost.

5. **Sandbox-invocation logging surface.** On-disk only via
   `meta.json` + `result.json` in the per-invocation scratch dir.
   No DB writes in v0.4. Defers the first schema change since v0.1.
   A `sandbox_invocations` table can land later when a real query
   need surfaces.

6. **Path-escape error mapping.** A `SourceFile.relative_path`
   containing `..` (or otherwise resolving outside `/scratch/src/`)
   maps to the existing v0.2 `input_validation_failed` error code.
   No `docs/tool-contracts.md` change. Path escape is technically
   an input-validation failure, so the umbrella fits.

7. **Seccomp / AppArmor / SELinux.** Deferred to a follow-on
   hardening milestone after Hunt and Validate consume the
   sandbox. ARCHITECTURE.md § Sandbox doesn't mandate them; the
   existing constraints (`--network=none`, `--cap-drop=ALL`,
   `--read-only` root, tmpfs scratch, resource caps, drop to
   `nobody:nogroup`) cover the v1 threat model.

## ARCHITECTURE.md amendment (proposed; lands in implementation commit)

Under § Sandbox, append after the "Firejail fallback" paragraph:

> **v0.4 scope note:** The first sandbox plumbing implementation
> ships both Docker (primary) and Firejail (fallback) backends,
> enforces the prescriptive constraints above verbatim
> (`--network=none`, read-only root, tmpfs `/scratch`, `/repo`
> read-only, `--memory=2g --cpus=2 --pids-limit=256`,
> `--cap-drop=ALL`, wall-clock timeout, 10 MB stdout/stderr caps),
> and selects between backends via auto-detection (Docker → else
> Firejail → else `sandbox_unavailable`). Backend selection is
> cached for the process lifetime.
>
> Hunt and Validate do **not** gain the `compile_and_run` tool in
> v0.4 — wiring the tool into their scopes is a follow-on milestone
> that consumes this layer. Until then, Hunt continues to operate
> with the v0.3 tool set and the prompt-level
> `confidence='confirmed'` prohibition stays in place.
>
> **Deferred to later milestones:** libFuzzer/AFL integration
> (`run_fuzzer` tool surface); seccomp / AppArmor / SELinux
> profiles; pre-built / pre-pushed language images; image-digest
> pinning with rotation policy; SBOM emission; the SSRF
> controlled-loopback HTTP fixture (policy plumbing is in v0.4,
> the fixture itself ships with the SSRF attack-class prompt).
> None of these weaken the constraints above; all are additive
> hardening or ergonomics.
>
> **Firejail toolchain caveat:** Firejail does not provide
> per-language filesystem images. When Firejail is the active
> backend, the host must have the language toolchain installed
> for the languages the run exercises — a deliberate trade-off
> for environments where Docker is unavailable (rootless desktops,
> locked-down CI). Operators on Firejail-only hosts get a clear
> error if a required toolchain is missing.

These are scope clarifications, not behaviour changes. The
prescriptive constraints under § Sandbox remain the prescriptive
v1 spec and are not relaxed.
