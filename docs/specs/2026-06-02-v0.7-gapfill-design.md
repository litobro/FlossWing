## Context

v0.2 shipped Recon end-to-end; v0.3 shipped Hunt end-to-end. After a
successful `flosswing scan <path>` run today, the state DB holds one
`recon_artifacts` row, ≥1 terminal `hunt_tasks` rows, ≥1 `findings`
rows, and matching `agent_sessions` rows. Validate is the next
prescribed pipeline stage, but Gapfill is the smallest remaining v1
stage by far and is useful in isolation — it produces additional Hunt
tasks the operator can re-run without Validate's machinery first.

**Milestone numbering is provisional.** "v0.7" was chosen by the
operating agent to leave numeric room for Validate / Dedupe / Trace
specs landing earlier; if any of those slip, this filename moves with
it. The slice itself is independent of that ordering decision.

This spec covers the **smallest vertical slice that proves Gapfill
works end-to-end**. Gapfill in v1 is the simplest agent stage per
`ARCHITECTURE.md § Stage 4`: one sequential agent, runs once per run,
reads the existing Recon artifact and Hunt task log, queues at most a
small number of additional Hunt tasks, then stops. No new SQLAlchemy
models, no schema change, no sandbox needs.

Chain to consult:

- `docs/specs/2026-05-25-v0.2-recon-plumbing-design.md` — Recon plumbing
- `docs/specs/2026-06-02-v0.3-hunt-plumbing-design.md` — Hunt plumbing

## Success criteria

A v0.7 implementation is done when **all** of the following are true:

1. `flosswing scan <path>` against `tests/corpus/v02_smoke/` runs
   Recon → Hunt → Gapfill in order, exits 0, and Gapfill records
   exactly one new `agent_sessions` row with `stage='gapfill'`,
   `outcome='completed'`, `input_tokens > 0`, `cost_usd > 0`.
2. After the run, the DB contains zero or ≥1 *additional* `hunt_tasks`
   rows with `source='gapfill'`. Zero is a valid outcome — Gapfill
   may conclude the original task set adequately covers a 50-file
   corpus. The count must not exceed the 20% cap (see § Budget cap).
3. CLI prints the run_id, the Recon summary (v0.2), the per-task Hunt
   summary (v0.3), and a Gapfill line:
   `gapfill → <outcome>, N new tasks queued, T tokens`.
4. New `hunt_tasks` from Gapfill remain `status='pending'` at end of
   run. **v0.7 does not auto-re-run Hunt** against them (per design
   decision #2; auto-re-pass deferred to a follow-on milestone). The
   CLI summary tells the operator they exist.
5. Refusal and budget paths are observable for Gapfill: a refused
   Gapfill records `outcome='refused'` with scrubbed refusal text;
   the run still completes (exit 0) as long as Recon and Hunt
   succeeded. **Gapfill failure is not run-fatal** — Recon + Hunt
   already produced the primary deliverable.
6. `ruff check .`, `mypy --strict flosswing`, and `pytest tests/unit`
   continue to pass; CI gains no new dependencies.
7. Schema is unchanged. `alembic upgrade head` round-trip CI passes.

## Scope

### In scope for v0.7

- New modules: `stages/gapfill.py`, `prompts/system/gapfill.md`,
  `tools/run_state.py`.
- `query_run_state` implementation per `docs/tool-contracts.md
  § run state`. Strict signature compliance.
- Tool list registered to Gapfill per the contract scope matrix:
  `read_file`, `list_dir`, `grep`, `query_findings`, `add_hunt_task`,
  `query_run_state`. Per design decision #6 (PR-walkthrough resolved):
  `query_findings` IS registered for Gapfill in v0.7 — v0.6 already
  shipped the tool implementation, so this is a one-line addition to
  the tool builder. Gapfill can see finding details (verdict, severity)
  not just per-task `findings_count`.
- Extension of `orchestrator.run_scan` to call `stages.gapfill.run(...)`
  after Hunt, only when Hunt's run-level outcome is `completed`.
- New CLI flag `--gapfill-token-budget INTEGER` (default 50000 per
  design decision #3; modest because Gapfill's input is small).
- New `config.py` field `gapfill_token_budget: int`.
- 20%-of-original-budget cap on new tasks queued (see § Budget cap).
  Enforced via the existing `add_hunt_task` `accepted=False` path.
- Refusal **detection and surfacing**. No retry, no rephrasing pass.
- Unit tests for `tools/run_state.py`, `stages/gapfill.py`, and the
  extended `add_hunt_task` cap path; integration-test extension.

### Out of scope for v0.7 (deferred)

- **Recursive Gapfill / multi-pass.** Binding per `ARCHITECTURE.md
  § Stage 4` ("Gapfill runs **once** per run; no recursive expansion
  in v1"). Do not propose this.
- **Automatic Hunt re-pass** over Gapfill-queued tasks. v0.7
  produces the tasks and stops; re-running Hunt on them is
  operator-driven (per design decision #2).
- Validate, Dedupe, Trace, Report stages.
- The v0.2 / v0.3 deferred list: `compile_and_run`, sandbox, symbol
  index, parallel Hunt, retry-on-refusal, schema-vs-models drift CI,
  additional attack-class prompt fragments.

### Explicit non-goals (per `ARCHITECTURE.md`)

Unchanged from v0.2 / v0.3: no daemon, no SaaS, no auto-disclosure,
no auto-patching, no telemetry, no cross-repo, no fuzzing framework,
no non-Anthropic providers, no write access to the target repo, no
recursive Gapfill.

## Architecture

### Module layout

```
flosswing/
  orchestrator.py         (existing — extend to call Gapfill)
  cli.py                  (existing — add --gapfill-token-budget)
  config.py               (existing — add gapfill_token_budget)

  stages/
    gapfill.py            (NEW)

  tools/
    findings.py           (existing — extend add_hunt_task cap path)
    run_state.py          (NEW)

  prompts/
    system/
      gapfill.md          (NEW)
```

No new top-level dependencies. No new SQLAlchemy models. No schema
migration. Stack remains exactly the list in `CLAUDE.md § Dependency
policy`.

`query_run_state` lives in its own module rather than
`tools/findings.py` because it reads across `runs`,
`recon_artifacts`, and `hunt_tasks` — none of which is the primary
subject of `findings.py`. When `query_findings` lands it can
co-locate with `query_run_state` in `tools/run_state.py`.

### Component responsibilities

**`stages/gapfill.py`** Gapfill-stage orchestration:

```python
async def run(
    *,
    run_id: str,
    repo: Path,
    cfg: Config,
    session_factory: SessionFactory,
    hunt_result: HuntStageResult,
) -> GapfillStageResult: ...
```

1. Compute the new-task cap (§ Budget cap). Stash on the per-session
   tool-registry context so `add_hunt_task` can enforce.
2. Load `prompts/system/gapfill.md`; compute SHA-256.
3. Build the Gapfill tool list per the scope matrix.
4. INSERT `agent_sessions` row (`stage='gapfill'`,
   `model=cfg.model_for('gapfill')`, partial fields).
5. Await `agent.runtime.run_session(...)` with
   `token_budget=cfg.gapfill_token_budget`.
6. Classify outcome; UPDATE the row.
7. Return:
   ```python
   class GapfillStageResult(BaseModel):
       outcome: Literal["completed", "refused", "budget_exceeded",
                        "errored", "skipped"]
       tasks_queued: int     # newly added rows with source='gapfill'
       cap: int              # the 20%-rule ceiling applied
   ```

**`tools/run_state.py`** Implements `query_run_state` exactly per the
contract. Reads the `runs` row, projects the `recon_artifacts` row
into `RecordReconArtifactInput` shape, projects each `hunt_tasks`
row into `HuntTaskSummary`. Computes `budget_used` as the sum of
`input_tokens + output_tokens` across `agent_sessions` for the run;
`budget_remaining` is best-effort against any total estimate the
config carries (else `0`). Read-only.

**`tools/findings.py` extension** to `add_hunt_task`: **no signature
change**. The cap check consults `ctx.gapfill_new_task_cap` when
that key is present on the registry context. When the cap is hit,
returns `accepted=False, reason="gapfill_cap_reached"` — same shape
as the contract's existing `budget_exhausted` outcome, distinguished
in `reason` for operator-output clarity.

**`orchestrator.run_scan` extension**:

After Hunt returns:

```python
if hunt_result.tasks_succeeded >= 1:
    gapfill_result = await stages.gapfill.run(
        run_id=run_id, repo=repo, cfg=cfg,
        session_factory=session_factory, hunt_result=hunt_result,
    )
else:
    gapfill_result = GapfillStageResult.skipped()
```

Run finalization rule additions:

- Hunt processed ≥1 tasks AND zero succeeded → `runs.status='errored'`,
  exit 1; Gapfill skipped (unchanged from v0.3).
- Hunt processed ≥1 tasks AND ≥1 succeeded → run is **completed**
  **regardless of Gapfill outcome**. Gapfill failure is logged and
  audited but does not flip the run to errored. Rationale: Recon +
  Hunt produced the primary deliverable; Gapfill is additive.

**`config.py` extension**: add `gapfill_token_budget: int`, default
50 000 input tokens. Same resolution chain as the v0.3 budget fields.
Gapfill's default model per `ARCHITECTURE.md § Agent runtime` is
`claude-sonnet-4-6`.

**`cli.py` extension**: add `--gapfill-token-budget INTEGER` (default
50000). Same convention as v0.3 budget flags.

### Gapfill system prompt — `prompts/system/gapfill.md`

Roughly:

- Target repo is **untrusted input** — same caveat as Recon / Hunt.
  Gapfill reads source via `read_file` / `grep` to verify coverage
  claims, so it's exposed.
- Lists the available tools. Explicit instruction on what
  `query_run_state` returns and how to interpret it.
- **Hard cap:** the agent may call `add_hunt_task` at most
  **`<cap>`** times this session (substituted at prompt-build time
  from the registry context). Tool layer enforces too.
- Success condition: call `query_run_state` once; optionally read /
  grep to verify coverage gaps; call `add_hunt_task` 0..cap times;
  stop. Recording zero new tasks is a valid outcome.
- Heuristic guidance on "thoroughly investigated" (per design
  decision #4 — agent judgment with this prompt guidance): a
  subsystem is "covered" if at least one
  non-`refused`/`errored` `hunt_tasks` row overlaps its path
  prefixes; an attack class is "under-represented" if the Recon
  architecture mentions a sink type for it but no task targets it.
  Phrased as guidance, not a rule — the agent retains judgment.
- Explicit instruction: **do not attempt recursive expansion**.
  Gapfill runs once per run.
- Refusal behaviour matches Hunt — refuse explicitly if input looks
  adversarial; we surface, don't punish.
- Hard size cap ~4 KB.

### Budget cap (20% rule)

Per `ARCHITECTURE.md § Stage 4`: "Outputs new Hunt tasks (capped at
20% of original budget)."

The ambiguity (resolved per design decision #1): "original budget"
could mean the `--budget N` concurrency cap from ARCH § Stage 2, the
original count of `source='recon'` tasks, or some token-spend figure.
v0.7 implements the **task-count interpretation**:

```
cap = max(1, ceil(0.20 * count(hunt_tasks WHERE source='recon')))
```

Rationale: most directly meaningful number in the existing schema;
matches the literal reading of "20% more tasks." On `v02_smoke`
(1 Recon task) this yields `cap=1`, matching the success criterion
("queues ≤1 additional task"). On a larger repo with 20 Recon
tasks, `cap=4`. The `--budget N` flag from ARCH § Stage 2 isn't
implemented yet (deferred from v0.3), so the concurrency
interpretation isn't reachable in v0.7.

The cap is computed once at Gapfill start, surfaced in the prompt,
and enforced by `add_hunt_task`.

## Data flow — one `flosswing scan <path>` run with v0.7

```
1. CLI → cli.scan(path, *_budget? flags)
2. config.resolve() → auth + budgets incl. gapfill_token_budget
3. orchestrator.run_scan(repo, cfg)
   a. ensure state DB; create run row & scratch dir
   b. await stages.recon.run(...)             [v0.2]
   c. if recon failed → finalize errored, exit 1
   d. await stages.hunt.run(...)              [v0.3]
   e. if zero Hunt tasks succeeded → finalize errored, exit 1
   f. compute gapfill_new_task_cap from hunt_tasks (source='recon')
   g. await stages.gapfill.run(...)
        - INSERT agent_sessions row (stage='gapfill')
        - load gapfill.md; substitute cap into prompt
        - await agent.runtime.run_session(...)
            → SDK spawns claude CLI subprocess
            → model calls query_run_state, optionally read_file/grep
            → model calls add_hunt_task 0..cap times
        - classify outcome; UPDATE agent_sessions row
   h. finalize: runs.status='completed' (Hunt succeeded; Gapfill
      outcome doesn't flip this), exit 0; print summary incl.
      Gapfill line
4. Process exits
```

Single-writer assumption still holds. Gapfill's session is
sequential and runs after Hunt's sessions complete.

## Error and refusal handling

Inherits all v0.2 / v0.3 categories. New categories from Gapfill:

| Condition | Where caught | What's stored | Run-level effect |
|---|---|---|---|
| Gapfill refused | `agent.runtime` | `agent_sessions.outcome='refused'`, `refusal_text=scrub(...)` | none — run still `completed` if Hunt succeeded |
| Gapfill over budget | orchestrator round check | `outcome='budget_exceeded'` | none |
| Gapfill errored | `agent.runtime` | `outcome='errored'`, `error_text=scrub(...)` | none |
| `add_hunt_task` beyond cap | `tools/findings.add_hunt_task` | nothing new; returns `accepted=False, reason="gapfill_cap_reached"` | agent receives structured response, can stop |
| Gapfill skipped (Hunt failed) | orchestrator | no `agent_sessions` row | run already errored upstream |

`errors.scrub()` continues to run on every string going to
`agent_sessions.refusal_text` / `error_text` and to stderr.

## Testing strategy

### Unit tests (in normal CI)

- `tests/unit/test_tools_run_state.py` — new:
  - happy path against in-memory DB seeded with runs +
    recon_artifacts + hunt_tasks rows
  - empty-architecture path (no recon_artifacts row):
    `recon_artifact is None`
  - budget arithmetic: sum of `input_tokens + output_tokens` across
    `agent_sessions` matches `budget_used`
- `tests/unit/test_tools_findings.py` — extend: with
  `ctx.gapfill_new_task_cap=2`, the third `add_hunt_task` call
  returns `accepted=False, reason="gapfill_cap_reached"`.
- `tests/unit/test_stages_gapfill.py` — new:
  - stubbed `runtime.run_session` returning canned successful /
    refused / errored sessions
  - assert `agent_sessions` row inserted and updated correctly
  - assert `GapfillStageResult.cap` matches the 20% arithmetic
  - assert Gapfill is **not** invoked when
    `hunt_result.tasks_succeeded == 0`

SDK still mocked at the `query()` / `tool()` boundary, not at HTTP
(per `CLAUDE.md § Testing`).

### Integration test (gated, NOT in CI)

Extend `tests/integration/test_recon_smoke.py`:

- `assert agent_sessions row exists with stage='gapfill'`, terminal
  outcome
- `assert COUNT(hunt_tasks WHERE source='gapfill') <= gapfill_cap`

A non-zero new-task count is **not** asserted — Gapfill may
legitimately conclude the existing `command_injection` task covers
the one-file CLI.

### CI changes

None. The existing pipeline covers everything.

## Design decisions (resolved 2026-06-04)

1. **20% rule interpretation.** Cap on Gapfill-queued tasks is
   `max(1, recon_task_count // 5)` — 20% of `source='recon'` task
   count, floor 1. Predictable, easy to test, lands in ARCH § Stage 4
   so future readers don't re-litigate. Alternative token-based and
   `--budget`-based interpretations rejected.

2. **Queue-and-stop; no auto-re-pass.** Gapfill writes new
   `hunt_tasks` rows with `source='gapfill'` and the scan exits.
   Auto-re-pass within the same run is deferred to a follow-on
   milestone (likely paired with `--resume <run_id>` work). The
   PR #4 ARCH scope note for v0.7 already reflects this stance.

3. **`--gapfill-token-budget` default = 50 000.** Gapfill's input
   is small (Recon artifact ~few KB, task summaries ≤ 1 KB each,
   selective spot-checks). Much smaller than Recon/Hunt's 200k and
   Validate's 100k.

4. **Coverage judgment: agent-driven with prompt guidance.**
   Gapfill prompt instructs the agent to call `query_run_state`,
   identify subsystems mentioned in the Recon artifact that received
   <2 hunt tasks, and propose new `add_hunt_task` calls for
   attack-class/subsystem combinations not yet covered. No
   orchestrator-side coverage rubric pre-computation; the prompt
   does the work. Stricter rubric is a future eval-driven follow-on.

5. **Run on `hunt_result.tasks_succeeded >= 1`.** Gapfill runs
   whenever Hunt completed at least one task cleanly, regardless of
   whether any findings were recorded. Zero-finding runs are exactly
   when Gapfill is most useful — propose new investigations that
   might surface what the initial Hunt missed.

6. **Register `query_findings` for Gapfill (operator-upsized).**
   The frozen scope matrix grants Gapfill `query_findings`. The
   original spec deferred it for minimum-slice; v0.6 has now shipped
   the implementation, so registering it for Gapfill is a one-line
   addition to the tool builder. Gapfill sees finding details
   (verdict, severity) instead of just per-task `findings_count`.

## ARCHITECTURE.md amendment (proposed; lands in implementation commit)

`ARCHITECTURE.md § Stage 4` is already terse and matches v0.7
closely. Proposed amendment is one paragraph clarifying Open
Question #2:

> **v0.7 scope note:** The first Gapfill plumbing implementation
> queues new Hunt tasks but does **not** auto-trigger a second
> Hunt pass against them within the same run. Newly-queued tasks
> sit in `status='pending'` for the operator's next invocation.
> Auto-re-pass within a single run is a follow-on milestone.

Consistent with the ARCH prose ("These re-enter the Hunt queue" is
compatible with "queued for the next invocation" — just
under-specified about timing). No amendment needed to the tool
allowlist; the matrix in `docs/tool-contracts.md` is authoritative.
